# Proteus v0.1 — "Hello, sequence" — Design

**Date:** 2026-05-16
**Status:** Approved, ready for implementation planning
**Source of truth:** `docs/SPECS.md` (Proteus spec v0.3). This document scopes the
v0.1 milestone (§20) and records the decisions and deviations made for it.

## 1. Goal

Ship the v0.1 milestone from `docs/SPECS.md` §20: a chat-only TUI that runs a
ReAct agent loop against one LLM provider, with filesystem tools, the
`fold.esmfold` tool, and the skills mechanism. No persistence.

### Acceptance criteria (from SPECS.md §20, v0.1)

1. `proteus` launches a TUI with a chat pane.
2. User can type `fold MAQVQLVESGGGLVQAGGSLRLSCAASGFTFSSYAMSWVRQAPGKGLEW` and the
   agent calls `fold.esmfold`, returning a PDB file path and pLDDT.
3. Switching to a local Ollama model via `/model` works.
4. `Ctrl+C` cancels mid-tool-call cleanly.
5. Smoke test passes in CI.

## 2. Scope

**In scope (strict v0.1):**

- Chat-only Bubble Tea TUI: status bar, scrolling chat pane, input/command bar,
  confirmation modal, model/provider picker overlay.
- ReAct agent loop running in a goroutine, communicating with the TUI via a
  `tea.Msg` bus channel.
- Two LLM providers: Anthropic (`anthropic.go`) and OpenAI-compatible
  (`openai.go`, which also serves Ollama / vLLM / LM Studio via `BaseURL`).
- A small hardcoded model registry; `/model` and `/provider` switching.
- Tools: `fs.read`, `fs.write`, `fs.edit`, `fs.bash`, `fold.esmfold`,
  `skills.list`, `skills.read`.
- One built-in skill: `filter-thresholds.md`.
- The §9 system prompt, embedded.
- Smoke test + CI.

**Out of scope (deferred to later milestones):**

- SQLite persistence, the `store` package (v0.2).
- Jobs / designs / wet-lab panels in the TUI (v0.2 / v0.4).
- Config files (`config.toml`, `models.toml`); v0.1 uses env vars + a hardcoded
  model list (v0.2+).
- Design tools (BindCraft, RFdiffusion, ProteinMPNN, …), scoring tools, the
  local/Modal backends (v0.2+).
- Knowledge tools, planning, the corpus (v0.3).
- Adaptyv integration (v0.4).
- Inline graphics, visualization, themes beyond light/dark, replay (v0.5).
- Context compaction (mentioned in SPECS.md §11; not required by v0.1).
- The Google provider (`google.go`); arrives in v0.3.

## 3. Architecture

### 3.1 Component map

```
cmd/proteus/main.go            Cobra root. Bare `proteus` -> TUI.
                               Subcommands: `tui`, `version`.

internal/version/version.go    Version string (ldflags-injectable).

internal/domain/types.go       MINIMAL domain types needed by the tool registry:
                               DesignID, JobID, ToolCallRef, Sequence, and a
                               trimmed Design (id, created, sequence,
                               structure_file, scores, provenance).

internal/llm/
  provider.go                  Provider interface + Chat/Stream types (SPECS §6.1).
  anthropic.go                 Anthropic provider.
  openai.go                    OpenAI-compatible provider (OpenAI + Ollama/vLLM/
                               LM Studio via BaseURL).
  registry.go                  Hardcoded model list; builds providers from env;
                               tracks active provider + model.

internal/agent/
  loop.go                      ReAct loop (SPECS §11). Goroutine; reads user
                               input from a channel; emits events on the bus.
  session.go                   Message history + system prompt. No compaction.
  prompts/system.md            SPECS §9 system prompt, embedded via go:embed.

internal/tools/
  registry.go                  Tool interface + Registry (SPECS §7.1).
  fs.go                        fs.read, fs.write, fs.edit, fs.bash.
  fold/esmfold.go              fold.esmfold tool.

internal/skills/
  loader.go                    Embeds builtin/; skills.list + skills.read tools.
  builtin/filter-thresholds.md The one v0.1 skill (SPECS §8.2).

internal/tui/
  app.go                       Root Bubble Tea Model: Update/View; wires agent,
                               registry, bus.
  chat.go                      Chat pane: viewport, streaming markdown via
                               glamour, collapsible tool traces.
  statusbar.go                 Top bar: project, model, provider, cost, elapsed.
  commandbar.go                Bottom input + slash-command bar.
  modal.go                     Confirmation modal (used by fs.bash denylist).
  theme.go                     Lipgloss styles; light/dark auto-detected.
```

### 3.2 Data flow

```
User types in commandbar
  -> app.Update routes plain text to the agent input channel
     (or handles a /slash command locally)
  -> Loop.Run goroutine:
       provider.StreamChat(req) -> stream of ChatEvent
       on tool_call events: Registry.Execute(ctx, name, input)
         - ctx is cancellable (Ctrl+C)
         - tool result appended to session, loop iterates
       on no tool calls: turn complete
  -> Loop emits tea.Msg values on the `bus` channel
     (text deltas, tool-call traces, tool results, done, error)
  -> app.Update receives bus messages -> chat pane re-renders
```

### 3.3 Concurrency & cancellation

- The agent `Loop` runs in its own goroutine. The TUI never blocks on the LLM.
- The TUI owns a `context.Context` per turn. `Ctrl+C` cancels it.
- `Registry.Execute` and every `Tool.Execute` accept `ctx` and must abort
  promptly when it is cancelled (HTTP requests use the context; `fs.bash`
  kills its child process). This satisfies acceptance criterion 4.
- Communication TUI <-> Loop:
  - TUI -> Loop: a buffered `chan string` for user input.
  - Loop -> TUI: the `bus chan tea.Msg`, drained by a Bubble Tea command that
    re-subscribes after each message.

### 3.4 Workspace

`fold.esmfold` is the only component that writes to disk. PDB files go to a
workspace directory:

- `PROTEUS_HOME` env var, default `~/proteus`.
- Workspace: `$PROTEUS_HOME/projects/default/designs/`.
- Created lazily on first write. The `save_as` input is resolved relative to
  the workspace; paths escaping the workspace are rejected.

## 5. Tool contracts (v0.1)

### 5.1 Filesystem & shell (`internal/tools/fs.go`)

| Tool | Input | Confirmation |
|---|---|---|
| `fs.read` | `{path}` | No |
| `fs.write` | `{path, content}` | No |
| `fs.edit` | `{path, old, new}` (exact string replace) | No |
| `fs.bash` | `{command, timeout_s?}` | Yes if command matches denylist |

- All paths resolve within the project workspace; escapes are rejected.
- `fs.bash`: 60s default timeout; `PATH` restricted to the allowlist
  (`ls cat grep sed awk jq python3 git curl wget`); denylist (`rm -rf`, `dd`,
  `mkfs`, `sudo`) triggers a confirmation modal.

### 5.2 `fold.esmfold` (`internal/tools/fold/esmfold.go`)

- Input schema (SPECS §7.2.2):
  `{ sequence: string /^[ACDEFGHIKLMNPQRSTVWY]+$/, save_as?: string }`.
- POST the raw sequence as the request body to
  `https://api.esmatlas.com/foldSequence/v1/pdb/`. No auth. Response is PDB text.
- Save the PDB to the workspace (`save_as`, else `designs/d_NNNN.pdb`).
- Parse pLDDT from the PDB B-factor column (ESM Atlas writes per-residue pLDDT
  there): `plddt_mean`, `plddt_min` over CA atoms.
- Output (SPECS §7.2.2):
  `{ design_id, structure_file, metrics: {plddt_mean, plddt_min}, elapsed_s }`.
- No confirmation; estimated cost $0; estimated duration ~15s.

### 5.3 Skills (`internal/skills/loader.go`)

- `skills.list` -> names + one-line descriptions of embedded skills.
- `skills.read` -> `{name}` returns the full markdown of one skill.
- v0.1 ships one built-in skill: `filter-thresholds.md`.

## 6. LLM providers

- `provider.go`: the §6.1 interface and types, copied verbatim, **plus** two
  fields added to `ChatEvent`: `Usage llm.Usage` and `StopReason string`. SPECS
  §11's loop reads `ev.Usage` / `ev.StopReason` on the `done` event but §6.1's
  struct omits them; this reconciles the inconsistency.
- `anthropic.go`: Anthropic provider. Tool use via native tool blocks.
- `openai.go`: OpenAI-compatible provider. A `BaseURL` field selects OpenAI,
  Ollama (`http://localhost:11434/v1`), vLLM, or LM Studio.
- `StreamChat` for v0.1 wraps a blocking `Chat()` call and emits the full
  response as one `text_delta` followed by `done`. The interface is the real
  streaming interface, so true token streaming can replace the wrapper later
  without touching the loop or TUI.
- `registry.go`: a small hardcoded list of `ModelDescriptor`s (Claude Opus 4.7,
  Claude Sonnet 4.6, one OpenAI model, one Ollama model). Default provider:
  `anthropic` if `ANTHROPIC_API_KEY` is set, else `ollama` if the local server
  responds, else the user is prompted. `/model` and `/provider` switch the
  active selection at runtime.

## 7. TUI

- Layout: chat-only. Status bar on top, chat pane in the middle, input/command
  bar at the bottom. The full multi-panel layout (SPECS §10.2) is v0.2+.
- Slash commands implemented in v0.1: `/model`, `/provider`, `/clear`, `/help`,
  `/quit`. Commands for not-yet-built subsystems (`/jobs`, `/designs`, `/plan`,
  `/lab`, `/export`, `/cost`, `/project`, `/skills`) print a short
  "available in a later milestone" notice rather than erroring.
- Keybindings: `Ctrl+C` cancel, `Ctrl+D` quit, `Esc` close modal/overlay,
  `Enter` send, arrows scroll the chat pane.
- Theme: light/dark auto-detected from terminal background (Lipgloss adaptive
  colors), using the status colors from SPECS §10.5.
- Streaming markdown rendered with `glamour`. Tool calls render as collapsible
  traces. Inline graphics are out of scope (v0.5).

## 8. Decisions & deliberate deviations from SPECS.md

1. **Dependency versions.** SPECS §3.2 pins versions (`anthropic-sdk-go v0.3.0`,
   `openai-go v0.1.0-beta.6`, `bubbletea v0.27.0`, …) that are likely stale.
   Actual versions are resolved with `go mod tidy`; the **provider/TUI adapter
   code** flexes to the real SDK APIs. Our own interface types (`Provider`,
   `ChatRequest`, `Tool`, …) stay exactly as specced.
2. **`ChatEvent` gains `Usage` and `StopReason`.** See §6 above.
3. **No context compaction.** SPECS §11 mentions it; v0.1's milestone does not
   require it. Deferred.
4. **Smoke test does not assert persistence or build a `Design`.** SPECS §18.2's
   smoke test asserts "session is persisted", but v0.1 has no persistence. The
   v0.1 smoke test instead drives the real agent loop with a mock LLM that calls
   `fold.esmfold` once against a fixture HTTP server, and asserts the agentic
   flow produces a fold result reporting `pLDDT` (via the tool result on the
   bus). It does NOT construct a `domain.Design`: SPECS §7.2.2 fixes
   `fold.esmfold`'s output as a flat `{design_id, structure_file, metrics,
   elapsed_s}` JSON object, and with no persistence in v0.1 there is no `Design`
   record to assert on. The `domain.Design` type is defined now (a forward
   declaration used by later code), but its construction and persistence are
   deferred to v0.2. The persistence assertion returns in v0.2.
5. **`fs.bash` allowlist drops `conda`.** SPECS §16.2 lists `conda`, but spec
   v0.3 replaced conda with uv project-wide. v0.1's allowlist:
   `ls cat grep sed awk jq python3 git curl wget`.
6. **Minimal domain types.** Only the subset in §3.1 is created. The full SPECS
   §4 type set arrives with v0.2 (when persistence needs it).
7. **Repo & CI.** `git init` locally; a GitHub Actions workflow runs
   `go build ./...` + `go test ./...`. Tests are offline (fixture-backed); no
   git remote is configured unless requested.

## 9. Testing

| Layer | Test |
|---|---|
| Domain types | Unit tests for `Sequence` and pLDDT parsing helpers. |
| `fold.esmfold` | Recorded-fixture test (a saved ESM Atlas PDB response) + an env-gated live test (`LIVE_E2E=1`). |
| LLM providers | Mock-based unit tests; live tests gated by `LIVE_LLM_TESTS=1`. |
| Tool registry | Unit tests for register / dispatch / schema. |
| Smoke test | `internal/agent/smoke_test.go`: mock LLM -> agent loop -> `fold.esmfold` (fixture HTTP server) -> assert the tool result reports `pLDDT`. This is the CI gate. |

CI: `.github/workflows/ci.yml` runs `go vet`, `go build ./...`, `go test ./...`
on push. All CI tests run offline.
