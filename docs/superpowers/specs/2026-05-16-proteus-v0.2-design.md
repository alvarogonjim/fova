# Proteus v0.2 — "Real designs" — Design

**Date:** 2026-05-16
**Status:** Approved-pending-review, ready for per-sub-project planning
**Source of truth:** `docs/SPECS.md` (Proteus spec v0.3), the v0.2 milestone (§20).
**Predecessor:** v0.1 "Hello, sequence" — complete and merged (tag `v0.1.0`).

## 1. Goal

Deliver the entire v0.2 milestone from `docs/SPECS.md` §20: SQLite persistence,
an async jobs system, the uv-based local tool installer, the Modal backend, and
the BindCraft / RFdiffusion / ProteinMPNN design tools plus ipSAE scoring — with
the jobs and designs panels in the TUI.

## 2. Scope and decomposition

v0.2 spans ~8 subsystems — far too large for one spec → plan → build cycle. It
is decomposed into **6 sub-projects (SP1–SP6)**. This document is the single
milestone design; **each sub-project then gets its own implementation plan**
(`docs/superpowers/plans/…`) and its own subagent-driven build cycle. All six
will be delivered.

| SP | Name | Delivers | Depends on |
|---|---|---|---|
| SP1 | Persistence foundation | Full `internal/domain` types (§4) + `internal/store/` (SQLite) | — |
| SP2 | Jobs system | `internal/jobs/` async manager + `internal/tools/jobs/` tools | SP1 |
| SP3 | Local backend (uv installer) | `internal/backends/local/` + `proteus install/uninstall/list tools/doctor` | — |
| SP4 | Modal backend | `internal/backends/modal/` + `proteus modal deploy` | — |
| SP5 | Design & scoring tools | `internal/tools/design/` + `internal/tools/score/` | SP1, SP2, SP3, SP4 |
| SP6 | TUI panels | jobs panel + designs panel; chat-only → multi-panel layout | SP1, SP2 |

**Build order:** SP1 → SP2 → SP3 → SP4 → SP5 → SP6. SP3 and SP4 are
self-contained and may be reordered; SP5 needs both backends; SP6 needs only
SP1+SP2 and is placed last so the panels are built against real data.

## 3. Sub-project designs

### SP1 — Persistence foundation

**Files:**
- `internal/domain/types.go` — expand the v0.1 minimal subset to the **full §4
  type set**: identifiers (`PlanID`, `SessionID`, `ProjectID`, `ExperimentID`),
  `Application` + consts, `JobKind`/`JobStatus` + consts, `ResidueRef`,
  `PDBReference`, the full `Design` (ProjectID, PlanID, Origin, Application,
  Sequence, StructureFile, Scores, LabResults, Provenance, Tags, Notes),
  `DesignOrigin` + consts, `FilterConfig` (with `MinIPSAE`), `DesignScore`,
  `DesignPlan`, `PaperRef`, `Job`, `Experiment`, `ExperimentResult`, `Session`,
  `Message`, `ToolCall`. Copied verbatim from §4.
- `internal/store/schema.sql` — the §5 DDL, embedded via `go:embed`.
- `internal/store/store.go` — open `<workspace>/workspace.db` via
  `modernc.org/sqlite` (pure-Go, no CGo); apply `schema.sql` idempotently on
  open; ensure a `default` project row exists.
- `internal/store/{designs,jobs,sessions,plans,experiments}.go` — typed CRUD
  per table. Rows that hold a JSON `body`/`input`/`output` column marshal the
  corresponding domain struct.

**Wiring:** `cmd/proteus/main.go` opens the store at startup and passes it to
the TUI/agent; the agent session is persisted to `sessions`/`messages`.

**Acceptance:** designs and sessions written in one run are present after a
restart (criterion 7, persistence half). Store CRUD is unit-tested against a
temp DB.

### SP2 — Jobs system

The §3.3 layout has no home for the async job *manager* (only the agent-facing
`internal/tools/jobs/` tools). **New package `internal/jobs/`** is added — a
justified addition.

**Files:**
- `internal/jobs/manager.go` — `Manager` owns running jobs and persists every
  state change to the `jobs` table.
  - `Submit(ctx, spec) domain.JobID` — spec carries `kind`, `tool`, `backend`,
    `input`, and a `run func(ctx, progress func(float64)) (output []byte, err error)`
    closure. `Submit` writes a `queued` row, spawns a goroutine that flips it to
    `running`, calls `run`, then records `succeeded`/`failed` with output/error.
  - `Status(id)`, `Result(id)` — **non-blocking**: return the current persisted
    `Job` (and its result if finished). Callers do not block for hours.
  - `Cancel(id)` — cancels the job's context; the `run` closure must honour it
    (kill the subprocess / abort the Modal call). Best-effort (criterion 9).
  - On startup, any job still `running` in the DB (crash leftover) is marked
    `failed` with an "interrupted" error.
- `internal/tools/jobs/{list,status,cancel,result}.go` — the four `jobs.*`
  agent tools (§7.2.9), thin wrappers over `Manager` + store queries.

**Async agent flow:** a long-running tool's `Execute` calls `Manager.Submit`
and returns `tools.Result{JobID: id, Display: "started job <id>, ETA <d>"}`
immediately. The agent reports the job and ends the turn; the user watches the
jobs panel; a later turn calls `jobs.result` to collect output. `jobs.result`
on an unfinished job returns "still running, ETA …".

**Acceptance:** criteria 8 (status/ETA surfaced) and 9 (cancellation).

### SP3 — Local backend (uv installer)

**Files (`internal/backends/local/`):**
- `tools.toml` — the §13.1.1 install recipes (bindcraft, rfdiffusion,
  proteinmpnn, ligandmpnn, rfantibody, ipsae, …), embedded via `go:embed`.
- `registry.go` — parse `tools.toml` into a tool registry; template-expand
  `{{ }}` / `${PROTEUS_HOME}` placeholders.
- `uv.go` — ensure `uv` itself is installed (bootstrap via the official
  installer if absent).
- `installer.go` — run a tool's `install_steps` in order in its `install_dir`;
  fetch `extra_data` (e.g. AlphaFold weights); on failure, the error names the
  failing step (criterion 4).
- `runner.go` — invoke an installed tool via its `run_command` for a job
  (writes the request JSON, runs the venv command, captures output).
- `doctor.go` — report per-tool install status, disk, uv presence (§13.1.3).

**CLI (in `cmd/proteus/`):** `proteus install <tool>` (+ `--all`, `--force`,
`--dry-run`), `proteus uninstall <tool>`, `proteus list tools`, `proteus doctor`.

**Acceptance:** criteria 1–4. **These require a real GPU machine, multi-GB
downloads, and minutes-to-hours of compute — they are MANUAL checks** (see §5).
Automated tests cover `tools.toml` parsing, placeholder expansion, doctor logic,
`--dry-run` command rendering, and the install-step sequencer with a stubbed
command runner.

### SP4 — Modal backend

**Files (`internal/backends/modal/`):**
- `functions.py` — a Modal app defining GPU functions that run the design tools
  (bindcraft/rfdiffusion/proteinmpnn) and ipSAE on Modal infrastructure. The
  user deploys it; Proteus does not host it.
- `client.go` — Go client that invokes the deployed Modal functions and maps
  results to the same schema the local backend produces.

**CLI:** `proteus modal deploy` — shells out to `modal deploy functions.py`.

**Acceptance:** criterion 5. **Requires a Modal account (`MODAL_TOKEN_ID`/
`MODAL_TOKEN_SECRET`) — MANUAL check.** Automated tests cover client
request/response mapping against an httptest stub.

### SP5 — Design & scoring tools

**Files:**
- `internal/tools/design/{bindcraft,rfdiffusion,proteinmpnn}.go` — each
  implements `tools.Tool`. `Execute` validates input, builds the request,
  submits an async job to the `jobs.Manager` with a `run` closure that dispatches
  to the local `runner` or the Modal `client` per the project's
  `compute_backend`. Returns `Result{JobID}` immediately. The job goroutine
  parses tool output into `domain.Design` rows in the store.
- `internal/tools/score/metrics.go` — parse pLDDT / pAE / ipTM from prediction
  outputs.
- `internal/tools/score/ipsae.go` — run the installed `ipsae` tool on a
  predicted complex (the primary interface metric).
- `internal/tools/score/filter.go` — apply `FilterConfig` thresholds and rank a
  shortlist by ipSAE first (§8.2 `filter-thresholds.md`).

**Backend abstraction:** a small `backend` interface (`Run(ctx, tool, input)
(output, error)`) with `local` and `modal` implementations, so a design tool's
job closure is backend-agnostic — guaranteeing criterion 10 (identical output
schema regardless of `compute_backend`).

**Acceptance:** criteria 6 and 10. **A real 100-binder BindCraft run is a MANUAL
check** (GPU + installed tools). Automated tests cover the `Execute` →
job-submission flow, the backend abstraction, score parsing, and filtering —
all with fixture outputs and a stubbed backend.

### SP6 — TUI panels

**Files:**
- `internal/tui/jobs.go` — `jobsModel`: the jobs panel (⟳/✓ rows, tool, id,
  ETA/elapsed), fed from the store + `jobs.Manager`.
- `internal/tui/designs.go` — `designsModel`: the designs panel, a table with
  the `ID pLDDT ipSAE ipTM Lab` columns (§10.2), fed from the store.
- `internal/tui/app.go` — expand the v0.1 chat-only layout to the §10.2
  multi-panel layout (chat left, jobs/designs/wet-lab stacked right); `Tab`
  cycles focus; collapse to chat-only below 100 columns.

**Live updates:** the `jobs.Manager` emits a bus message on every job state
change; `app.go` refreshes the panels. Designs appear as the store gains rows.

**Acceptance:** criteria 7 (designs panel with ipSAE) and 8 (jobs panel).

## 4. Key decisions & deliberate deviations

1. **Config — minimal per-project setting.** `compute_backend` (default
   `"local"`) and the few v0.2 settings live in a per-project record, overridable
   by env var (`PROTEUS_COMPUTE_BACKEND`). The full §14 config system
   (`config.toml`, `models.toml`, koanf, `internal/config/`) stays deferred —
   it is not in v0.2's Implements list.
2. **Projects — single `default` project.** The `projects` table and all
   `project_id` foreign keys are created (schema stays forward-compatible), but
   v0.2 auto-creates and uses exactly one `default` project. `proteus project
   *` CLI and `/project` are deferred (§15, not v0.2's list).
3. **Migrations — idempotent schema application.** `schema.sql` uses
   `CREATE TABLE IF NOT EXISTS`; it is applied on every startup. No versioned
   migration framework — YAGNI for v0.2.
4. **ipSAE is a `tools.toml` entry**, installed like any other tool (`proteus
   install ipsae`, shown by `proteus doctor`) — criteria 2 and 3 only make sense
   for a registry tool. It is lightweight (criterion 2: <60 s).
5. **New package `internal/jobs/`** holds the async job manager (the §3.3 layout
   only placed the agent-facing `jobs.*` tools). Justified addition.
6. **Async, non-blocking jobs.** Design jobs run for minutes-to-hours; the agent
   never blocks a turn on them. `jobs.status`/`jobs.result` return current
   state; the jobs panel shows live progress.
7. **`domain.Design` is now constructed** (by SP5's design jobs and persisted by
   SP1) — closing the v0.1 deferral noted in the v0.1 design doc §8.4.
8. **Deferred to later milestones:** full config/models system, project-
   management commands, cost tracking (`internal/cost/`), credential keyring,
   knowledge tools, Adaptyv, visualization, inline graphics, real LLM streaming.

## 5. Testing & verification strategy

The v0.1 pattern extends: **all CI tests are offline and deterministic**;
external/heavy operations are exercised against stubs/fixtures, with real runs
gated behind env vars.

**Automated (CI — `go test ./...`):** domain types; store CRUD against a temp
SQLite DB; the job manager state machine and cancellation; `jobs.*` tools;
`tools.toml` parsing and placeholder expansion; doctor logic; installer step
sequencing with a stubbed command runner; `--dry-run` output; score parsing and
filtering with fixture prediction outputs; design-tool `Execute` →
job-submission flow with a stubbed backend; Modal client mapping against an
httptest stub; TUI panel rendering.

**MANUAL (cannot run in CI — need a real GPU box, multi-GB downloads, or
accounts):** criteria 1–4 (`proteus doctor`/`install` of real tools), criterion
5 (`proteus modal deploy`), and a real end-to-end design run for criterion 6.
These are documented manual checks; the code is built to spec and unit/fixture-
tested, but a true GPU run is the user's verification step.

This constraint is inherent to v0.2 ("Real designs" needs real GPUs) and is
called out so it is not a surprise at acceptance time.

## 7. Execution approach

For each sub-project SP1 → SP6, in order: invoke `writing-plans` to produce a
detailed implementation plan, execute it via subagent-driven development
(implementer + spec/quality review per task), finish the branch. Each
sub-project is a self-contained, tested increment. This design doc is the
shared reference for all six.
