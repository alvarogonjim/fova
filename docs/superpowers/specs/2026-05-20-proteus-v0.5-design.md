# Proteus — v0.5 "Polish" — Design

**Date:** 2026-05-20
**Status:** Approved, ready for per-SP planning
**Milestone:** SPECS §20 v0.5 "Polish"
**Predecessors:** v0.4 tagged; SPECS §14 configuration system complete (SP1/SP2/SP3 merged).

## 1. Goal & scope

Deliver every v0.5 SPECS §20 line. The milestone spans **seven independent
subsystems**, none of which exist yet (the SPECS tools table marks
`viz.metric_plot` as v0.3, but it was never implemented — its build lands here).

| SP | Subsystem | Package(s) | Acceptance |
|---|---|---|---|
| A | Theming + keybindings | `internal/tui` | §20 #4 |
| B | Inline graphics | `internal/tui/graphics.go` | §20 #1 |
| C | Visualisation tools | `internal/tools/viz/` | §20 #1 (rendering for inline display) |
| D | Knowledge tools | `internal/tools/knowledge/` | — |
| E | Biosecurity | `internal/safety/` | §20 #3 |
| F | Replay | `cmd/proteus/replay.go` | §20 #2 |
| G | Release engineering | `.github/`, `.goreleaser.yaml`, `install.sh` | §20 #5 |

The seven SPs share **no source files** except `cmd/proteus/main.go`
(tool/command registration). Conflicts there are restricted to sibling
`registry.Register(...)` lines and a single `replay` cobra command and can be
resolved by sequential merging into `master` in alphabetical order (A → G).
SPs are designed to be **parallel-implementable** by independent Opus agents
in separate worktrees.

## 2. SP-A — Theming and the full keybinding set

**Goal:** consume `[ui].theme` so a user can force light or dark independently
of the terminal background; expose `/theme`; complete the SPECS §10.4
keybinding set.

**Design.** `lipgloss` exposes `lipgloss.SetHasDarkBackground(bool)` — calling
it overrides the auto-detection that drives `lipgloss.AdaptiveColor`. SP-A
applies one of three modes at startup, derived from `cfg.UI.Theme`:

- `auto` (default) — leave detection alone; the current v0.4 behaviour.
- `light` — `lipgloss.SetHasDarkBackground(false)`.
- `dark` — `lipgloss.SetHasDarkBackground(true)`.

The existing `DefaultPalette` (an `AdaptiveColor` set) needs no changes — its
`Light` and `Dark` arms are already correct; SP-A just forces which arm is
selected. A `/theme <auto|light|dark>` slash command applies the same call at
runtime and writes the choice back to `~/.config/proteus/config.toml`.

**Keybindings.** SPECS §10.4 lists seven keys; v0.4 ships them all. SP-A adds
the rest of a "modern TUI" keybinding set and a `/keys` help overlay listing
every binding:

- `PgUp`/`PgDown` — scroll the chat panel.
- `Home`/`End` — scroll-to-top / scroll-to-bottom of the chat.
- `Ctrl+L` — clear the chat (equivalent to `/clear`).
- `Ctrl+R` — reload `config.toml` and `models.toml` (no restart).
- `?` (in the input bar's command mode) — open the `/keys` overlay.

A `keybindings.go` file owns the table of `{key, action, description}` rows so
`/keys` reads from one source of truth.

**Files:** modify `internal/tui/app.go` (key routing, `/theme`, `/keys`,
`/reload`), `internal/tui/theme.go` (apply forced mode), `internal/config`
(add `SaveConfig` for `/theme` writeback). Create
`internal/tui/keybindings.go` and `internal/tui/keys.go` (the `/keys` overlay).

**Out of scope:** custom palettes (a single light/dark palette ships; user
palettes stay v1+).

## 3. SP-B — Inline graphics

**Goal:** `internal/tui/graphics.go` — detect the terminal, encode a PNG into
the active graphics protocol, expose `RenderImage(path string) (string, bool)`
returning the escape-sequence-wrapped image plus a "supported?" flag.

**Detection** (consumed at startup, cached):

- `KITTY_WINDOW_ID` set → Kitty.
- `TERM_PROGRAM == "iTerm.app"` or `LC_TERMINAL == "iTerm2"` → iTerm2.
- `TERM` contains `xterm-256color` and a SIXEL probe (writing `\x1b[c`, reading
  back the response) reports `4` in the attributes → Sixel.
- Otherwise → off (rendering returns `("", false)`).

`[ui].inline_graphics` (already parsed in SP2) overrides detection: `auto`
keeps the detection, any explicit value forces that protocol or `off`.

**Encoding.** A small per-protocol encoder writes the PNG bytes:

- **Kitty:** `\x1b_Ga=T,f=100;<base64-png>\x1b\\` (chunked at 4096 chars).
- **iTerm2:** `\x1b]1337;File=inline=1:<base64-png>\x07`.
- **Sixel:** call out to `img2sixel` if present (best-effort; Go-native sixel
  encoding is too much for v0.5).

**Wiring.** After a fold/design job that produces a structure file, the TUI
calls `viz.pymol_render` (SP-C) on the result, then `graphics.RenderImage()`
on the returned PNG, and appends the escape string as a new chat entry. The
existing `chatModel` is unchanged — the rendered string is just text.

**Files:** create `internal/tui/graphics.go`, `internal/tui/graphics_test.go`.
Modify `internal/tui/app.go` to detect once in `New` and render structure
images after job-done events.

**Out of scope:** terminal-resize re-render of the cached escape; transparent
backgrounds; animation.

## 4. SP-C — Visualisation tools (`internal/tools/viz/`)

Four tools, each a small `tools.Tool` implementation.

**`viz.metric_plot`** — PNG of score distributions. Input: a `results_id`
(produced by `score.metrics`) or a raw `{name: [...values]}` map. Output: a
PNG histogram per metric, side-by-side. Implemented with `gonum.org/v1/plot`
(no Python dependency).

**`viz.contact_map`** — PNG of inter-chain Cα–Cα contacts. Input: a path to a
PDB/CIF. Output: a heatmap PNG sized by min(N, 800)×min(M, 800) pixels with
greyscale gradient. The parser only needs Cα coordinates by chain — written
in-package, no external structure-IO library.

**`viz.ascii_structure`** — ASCII secondary-structure string for a PDB. Input
a PDB path. Output: per-chain `HHHHH-EEEEE---HHHHH` plus a SEQ line aligned
beneath. Computed from φ/ψ angles via a 30-line DSSP-lite (helix/strand/coil
only — accurate enough for the chat preview, not for production).

**`viz.pymol_render`** — PNG of a structure via PyMOL CLI:
`pymol -c -q -d "load <pdb>; bg_color white; orient; ray 600,600; png <out>"`.
The tool detects PyMOL via `exec.LookPath("pymol")` at construction; absent →
return a clear error at call time. No PyMOL → the inline-display wiring in
SP-B uses `ascii_structure` as the fallback.

All four tools persist the rendered PNG under
`~/proteus/projects/<project>/designs/<id>.png` and return the path in
`Provenance`.

**Files:** create `internal/tools/viz/{metric_plot,contact_map,ascii_structure,pymol_render}.go`
and matching `_test.go` files. Modify `cmd/proteus/main.go` to register the
four tools.

**Out of scope:** animations, interactive viewers, multi-state ensembles.

## 5. SP-D — Knowledge tools

**`knowledge.blast`** — NCBI BLAST URL API
(`https://blast.ncbi.nlm.nih.gov/Blast.cgi`). Submit `CMD=Put&PROGRAM=blastp&...`
with a sequence, parse the `RID` from the HTML reply, poll
`CMD=Get&RID=<id>&FORMAT_TYPE=JSON2_S` every 15 s until ready (cap 5 min),
return parsed hits (top-N accession, e-value, %identity). Stateful but
tool-local — no persistence beyond returning the results.

**`knowledge.local_pdfs`** — scan a `local_pdfs.dir` set in `config.toml`
(SP-D extends the `[knowledge]` section with one optional field), index every
PDF into a bleve index parallel to `knowledge.corpus`'s, and expose
`local_pdfs.{add, search, ask}` mirroring the `corpus` API. PDF text is
extracted with `github.com/ledongthuc/pdf` (pure Go).

**`knowledge.paperclip`** — optional MCP wrapper. Registered only when
`PAPERCLIP_TOKEN` is set in the environment. The tool acts as a thin
forwarder: `{action: "search", query: ...}` → POST to the Paperclip MCP
endpoint, return the response verbatim. No local state. The MCP transport is
plain HTTP+JSON (the same shape `mcp__*` adapters use).

**Files:** create `internal/tools/knowledge/{blast,local_pdfs,paperclip}.go`
plus tests. Extend `[knowledge]` in `internal/config/config.toml` with
`local_pdfs_dir = ""` (optional). Modify `cmd/proteus/main.go` to register the
three tools (paperclip conditionally).

**Out of scope:** automatic local-PDF refresh; Paperclip authentication flow
beyond the env-var token.

## 6. SP-E — Biosecurity

**Goal:** refuse `design.*` and `lab.submit_experiment` calls when the
*target* sequence matches a restricted-target list (SPECS §20 #3).

**Design.** `internal/safety/restricted_targets.go` exposes:

```go
type Refusal struct { Match string; Reason string }
func Check(seq string) (Refusal, bool) // (refusal, true=refuse)
```

The matcher is a substring/k-mer screen against a curated table of restricted
sequences (subset of the Australia Group / select-agents list — handful of
20–30 residue signatures of high-risk toxins; the v0.5 ship-list lives in
`internal/safety/restricted_targets.toml`, embedded). A match returns the
refusal reason and the agent surfaces it as a user-facing chat error and
declines the tool call.

**Wiring.** A `safety.Guard` is added to the `tools.Registry` (or attached to
the agent loop's pre-tool-call hook): before executing any tool whose name
matches `design.*` or `lab.submit_experiment`, the registry pulls the target
sequence from the input JSON (`target_sequence`, `target.sequence`, or the
sequences themselves for `lab.submit_experiment`) and runs `safety.Check`.
On refusal, the tool call is replaced by an error result and the agent sees a
clear `"refused by biosecurity guard: <reason>"` string.

**Files:** create `internal/safety/{restricted_targets.go,restricted_targets.toml,guard.go,*_test.go}`.
Modify `internal/tools/registry.go` to call the guard (or a small change in
`internal/agent/loop.go`'s `executeTool` — whichever path is the natural
choke-point). Modify `cmd/proteus/main.go` to wire the guard once.

**Out of scope:** distributed updates of the list; per-project overrides;
encrypted match tables.

## 7. SP-F — Replay (`proteus replay`)

**Goal:** `proteus replay <session.json>` reproduces a recorded session
deterministically (SPECS §20 #2).

**Design.** A session is already persisted to SQLite (`internal/store`,
`Session`/`Message`/`ToolCall` tables). SP-F adds two pieces:

1. **Export.** `proteus export <session-id> <out.json>` — writes the session
   as a normalised JSON document: ordered list of `{kind, payload}` events
   (user_msg, agent_delta, tool_start, tool_result, turn_done) preserving
   timestamps and recorded tool outputs.
2. **Replay.** `proteus replay <session.json>` — opens the TUI in
   "replay mode": the agent loop is bypassed; events are fed into the chat
   from the JSON, paced by their original timestamps (capped at 50 ms/event
   for a fast skim). No LLM or tool calls are made; the recorded outputs are
   shown directly. A footer line shows `replay 12/42 events`.

**Files:** create `cmd/proteus/replay.go`, `cmd/proteus/export.go`,
`internal/replay/{replay.go,format.go,*_test.go}`. Modify the TUI's
`Deps`/`New` with a `ReplayEvents []replay.Event` field that, when non-nil,
drives the model in place of an agent session.

**Out of scope:** re-executing tools live during replay; rewinding mid-replay;
diffing two sessions.

## 8. SP-G — Release engineering

**Goal:** single-binary releases for the five SPECS §20 target platforms,
plus a Homebrew tap and an install script (§20 #5).

**Design.**

- **`.goreleaser.yaml`** — five build matrix entries (darwin-amd64,
  darwin-arm64, linux-amd64, linux-arm64, windows-amd64), `CGO_ENABLED=0`,
  `-trimpath`, `-ldflags="-s -w -X .../version.version={{.Tag}}"`. Archives:
  `tar.gz` on Unix, `zip` on Windows. SHA256 checksum file.
- **GitHub Actions workflow** `.github/workflows/release.yml` — triggers on
  `tag` matching `v*`, runs `goreleaser release --clean`. A second workflow
  `.github/workflows/ci.yml` runs `go build / go test / go vet / gofmt -l` on
  every push (master) and pull-request.
- **`install.sh`** — `curl | sh` script: detect OS/arch, download the
  matching archive from the latest GitHub release, extract `proteus` to
  `~/.local/bin/`, print "Add `~/.local/bin` to PATH".
- **Homebrew tap** — a single `homebrew-tap/Formula/proteus.rb` file
  generated by goreleaser's `brews` block, pushed to the user's
  `<owner>/homebrew-tap` repo. A `formula_template.rb` lives in the repo for
  reference.

**Verification.** Locally: `goreleaser build --snapshot --clean` produces the
five binaries under `dist/` — the design verifies that on the dev box. A real
published release is out of scope here (it requires repo-admin / secrets to
publish to a GitHub release and push to a tap repo).

**Files:** create `.goreleaser.yaml`, `.github/workflows/release.yml`,
`.github/workflows/ci.yml`, `install.sh`, `homebrew-tap/Formula/proteus.rb`,
and a short `docs/RELEASING.md` walking through the tag→release process.

**Out of scope:** signed builds (cosign/notary); publishing to package
registries other than Homebrew; auto-update on launch.

## 9. Parallelism plan

After each SP has its plan written, dispatch them as concurrent worktree-
isolated implementation agents (Opus). Independent SPs (A, B, C, D, E, F, G —
no shared source files apart from `main.go` registration) can run in
parallel. Each worker creates a branch `feat/v0.5-sp-<letter>-<name>` off
`master` (verified with `git merge --ff-only master`); each commits and
merges back into `master` sequentially A→G. The `main.go` `buildRegistry`
diffs are sibling lines and resolve cleanly under `ort`; any conflict is
trivially fixed by appending both registrations.

## 10. Testing & acceptance

Per-SP tests are detailed in each SP's plan. The milestone-level acceptance
mirrors SPECS §20 (#1–#5). After all seven merges to `master`:

- `go build ./... && go test ./... && go vet ./... && gofmt -l` — clean.
- Bump `internal/version/version.go` to `0.5.0-dev`; tag `v0.5.0` after the
  user has verified one inline-graphics render in their own Kitty/iTerm2
  terminal (an environment we cannot exercise from here).

## 11. Out of scope (the whole milestone)

`viz.dssp` (a real DSSP wrapper — the ASCII version is a hack); video/GIF
output; multi-locale theme palettes; biosecurity refresh-from-server;
session-diffing; signed binaries; auto-update.
