# v0.6 — Design-tool agent UX gaps

**Date opened:** 2026-05-20
**Last updated:** 2026-05-20
**Status:** Living document — being extended as manual GB10 testing surfaces more issues
**Component:** `internal/backends/local/` design adapters, `internal/tools/design/` wrappers, `internal/jobs/`, `internal/agent/`
**Affects since:** v0.4 (when the design adapters first landed via SP1–3)

## Why this ticket exists

Manual end-to-end testing of v0.5 on GB10 hardware surfaced a cluster of related agent-UX gaps in the design-tool path. Each one in isolation is small; together they make the design loop unreliable in the realistic case where the agent — not the human — picks file paths and decides when a job is "stuck."

This document records each bug with reproduction, root cause, and proposed fix. The implementation will land as a single v0.6 sub-project once we stop adding to it.

---

## Bug 1 — Relative tool paths don't resolve against the project workspace root

### Summary

Every `design.*` tool in the local backend (`design.proteinmpnn`, `design.rfdiffusion`, `design.bindcraft`, `design.rfantibody`, `design.rfdiffusion2`, `design.ligandmpnn`, `design.chai2`) treats its `target` input as a literal filesystem path and validates it with `os.Stat`. Because Go's `os.Stat` resolves relative paths against the **process working directory**, an agent that passes `designs/d_0003.pdb` only succeeds when Proteus was launched from inside the project's workspace root. In any other case the call fails with `target … does not exist`, even when the file is right there in the active project's `designs/` directory.

The `fs.*` tools already do this correctly via `tools.SafeJoin(workspaceRoot, rel)`. The design adapters were written before that pattern existed.

### Reproduction

1. Launch Proteus from outside the project workspace, e.g. from the source tree:
   ```bash
   cd ~/Projects/proteus
   ./bin/proteus tui
   ```
   Status bar reports workspace `/home/gonjim/Projects/proteus`, but designs are persisted under `~/proteus/projects/default/designs/`.
2. Fold a sequence so a real PDB exists in the project's `designs/` directory.
3. Prompt: `Run design.proteinmpnn on designs/d_0003.pdb with num_designs=3.`
4. Observed: `job failed · 0% · error: design.proteinmpnn: target "designs/d_0003.pdb" does not exist`.
5. Expected: ProteinMPNN runs against the existing file.

### Root cause

`internal/backends/local/adapter_proteinmpnn.go` (and the sibling adapters) validate the target with a raw `os.Stat`:

```go
target := strings.TrimSpace(req.Target)
…
if info, err := os.Stat(target); err != nil || info.IsDir() {
    return nil, fmt.Errorf("design.proteinmpnn: target %q does not exist", target)
}
```

`adapter_bindcraft.go` and `adapter_rfdiffusion.go` follow the same pattern for their `target` / `starting_pdb` / `contigs` reference files. No adapter is aware of the project workspace root.

The design tool **wrappers** (`internal/tools/design/proteinmpnn.go` and siblings, registered in `cmd/proteus/main.go`) similarly take only `(mgr, backend, st)` — they have no `workspaceRoot` to resolve against, unlike `fold.NewESMFold(workspaceRoot)` and `tools.NewFSTools(workspaceRoot)`, which do.

### Proposed fix

Thread the project workspace root through the design tool wrappers and have them pre-resolve relative paths before handing the request to the backend. This keeps the adapters honest (they continue to operate on absolute paths and never need to know about projects) and matches the established pattern for `fs.*` and `fold.esmfold`.

1. **Tool wrappers** — extend each `New*Tool` constructor to accept `workspaceRoot`. Today `internal/tools/design/design.go` defines a shared `designTool` struct; either add the field there once or pass through each per-tool constructor.

2. **Path resolution helper** — add to `internal/tools/` (sibling to `SafeJoin`):

   ```go
   // ResolveWorkspacePath returns p unchanged if it's absolute, else joins it
   // onto root using SafeJoin (rejecting escapes). Empty p returns "".
   func ResolveWorkspacePath(root, p string) (string, error) { … }
   ```

   Use it in the design wrapper's `Execute` before calling `backend.Run(...)`. Inputs to rewrite include `target`, `starting_pdb`, `contigs_pdb`, `theozyme` — anywhere the tool schema declares a workspace-relative path.

3. **Wire it in `cmd/proteus/main.go`**: pass the existing `workspace` variable into the seven design constructors (it's already in scope and already passed to `NewFSTools` and `NewESMFold`).

4. **Adapters unchanged.** The `os.Stat` validation stays, but it now sees absolute paths and yields a clearer error in the genuine "file is missing" case.

5. **Docstrings on each design tool's `InputSchema`** updated to say the path is *workspace-relative* (matching `fs.read`'s description).

### Acceptance criteria

1. Launching `proteus tui` from any directory and prompting `design.proteinmpnn` with a workspace-relative path (`designs/d_0003.pdb`) succeeds when the file exists under the active project's workspace.
2. Absolute paths inside the workspace still work.
3. Absolute paths *outside* the workspace are rejected with `target "…" escapes the workspace` (same wording as `fs.read`).
4. Path-traversal attempts (`../../etc/passwd`) are rejected for every design tool.
5. New unit test in `internal/tools/design/` per wrapper covering: relative-path resolution, absolute-path passthrough, escape rejection, empty-path rejection — mirroring `internal/tools/fs_structure_test.go::TestFSReadStructureRejectsEscape`.
6. Existing adapter tests continue to pass unchanged (the adapter contract stays "absolute path in").
7. `go test ./...` and `go vet ./...` clean.

### Out of scope for Bug 1

- **Cross-project file access.** This keeps every design call confined to the active project's workspace.
- **Adapter-side workspace awareness.** The proposed fix keeps adapters thin; they still operate purely on absolute paths.
- **Fold adapters.** `fold.esmfold` already takes `workspaceRoot` for its *output* path. `fold.boltz2` / `fold.chai1` should be audited under the same ticket — fold-in acceptable if it doesn't blow up the diff.

### Effort estimate

~half a day; mostly mechanical (seven wrappers, one helper, seven test files).

## Bug 2 — Design adapters never stream output or progress to the job log

### Summary

Two visible symptoms have one root cause:

- **Empty job log files.** Every `design.proteinmpnn` job log under `~/proteus/logs/` is 0 bytes — both failed jobs and the cancelled one (`j_dd3735d5`, ran 2 min 16 s). Install jobs (`j_70ea8e06` ipsae, 676 bytes; `install-proteinmpnn-1779183994`, 1084 bytes) write their logs correctly. Design jobs do not.
- **Job progress is stuck at `0%`** for the entire run, then jumps straight to 100% on success. There is no visibility into liveness or stage.

Together these make the agent unable to tell a slow but progressing job from a hung one. In the observed run the agent polled `jobs.status` 10+ times, never saw progress move, and cancelled at 2 min 16 s — likely seconds before ProteinMPNN would have finished (CPU-mode for 129-aa lysozyme with 3 designs is in the ~2-3 min range).

### Reproduction

1. From the workspace root, with `CUDA_VISIBLE_DEVICES=` set:
   ```
   Run design.proteinmpnn on designs/d_0003.pdb with num_designs=3.
   ```
2. Watch the jobs panel — progress sits at `0%` indefinitely.
3. `cat ~/proteus/logs/<job-id>.log` while the job is running — empty file.
4. Cancel the job (or let it complete). Log remains empty on cancel; on success it contains only the final JSON `{"designs":[…]}` envelope.

### Root cause

1. **`bashRunner` uses `cmd.CombinedOutput()`**, which buffers all stdout+stderr in memory and returns it only when the process exits (`internal/backends/local/installer.go:19-26`).

   ```go
   func bashRunner(ctx context.Context, dir, command string) (string, error) {
       cmd := exec.CommandContext(ctx, "bash", "-c", command)
       if dir != "" { cmd.Dir = dir }
       out, err := cmd.CombinedOutput()
       return string(out), err
   }
   ```

2. **`AdapterEnv` carries no log writer or progress callback** (`internal/backends/local/adapter.go:20-25`):

   ```go
   type AdapterEnv struct {
       Recipe   ToolRecipe
       Run      CmdRunner
       WorkDir  string
       Registry *Registry
   }
   ```

   So even if an adapter wanted to stream progress, it has nowhere to send it.

3. **`localBackend.Run` writes the log only after the adapter returns** (`internal/backends/backend.go:24-29`):

   ```go
   func (b *localBackend) Run(ctx context.Context, tool string, input []byte, log io.Writer) ([]byte, error) {
       out, err := local.RunDesign(ctx, b.registry, tool, input)
       _, _ = log.Write(out)
       return out, err
   }
   ```

   When `RunDesign` returns `(nil, err)` (failure path) or hasn't returned yet (cancellation), nothing is ever written.

4. **The design tool's outer `Run` only ticks progress to `0.9` after the backend returns** (`internal/tools/design/design.go:67-79`). The `progress` callback is never threaded through to the adapter.

The install path differs because `installer.InstallLogged(ctx, name, log)` takes the log writer directly and writes each step's output to it as it goes — so install jobs get real, populated log files.

### Proposed fix

Thread a log writer and a progress callback all the way from the job manager into the adapter, and stream `bashRunner`'s output to that writer in real time.

1. **Change `CmdRunner` signature** to take a log writer:

   ```go
   type CmdRunner func(ctx context.Context, dir, command string, log io.Writer) (string, error)
   ```

2. **Update `bashRunner`** to tee live stdout+stderr to both an in-memory buffer (for the return value) and the log:

   ```go
   func bashRunner(ctx context.Context, dir, command string, log io.Writer) (string, error) {
       cmd := exec.CommandContext(ctx, "bash", "-c", command)
       if dir != "" { cmd.Dir = dir }
       var buf bytes.Buffer
       w := io.MultiWriter(&buf, log)
       cmd.Stdout = w
       cmd.Stderr = w
       if err := cmd.Run(); err != nil {
           return buf.String(), err
       }
       return buf.String(), nil
   }
   ```

3. **Extend `AdapterEnv`** with `Log io.Writer` and `Progress func(float64)`. Default values are `io.Discard` and a no-op, so tests don't have to provide them.

4. **Update `local.RunDesign` and `localBackend.Run`** to accept and thread `log io.Writer` and `progress func(float64)`. The `Backend` interface gains the `progress` parameter:

   ```go
   type Backend interface {
       Name() string
       Run(ctx context.Context, tool string, input []byte, log io.Writer, progress func(float64)) ([]byte, error)
   }
   ```

5. **Per-adapter progress ticks.** Each adapter knows its own stages — for ProteinMPNN:
   - `0.05` after target staging
   - `0.20` after `parse_multiple_chains.py` returns
   - `0.95` after `protein_mpnn_run.py` returns
   - The outer design tool moves to `1.0` on persistence.

   For BindCraft and RFdiffusion the stage list is different; the principle is the same.

6. **The Modal backend** mirrors the change — its `Run` signature gains the same parameters; it can write the returned payload to the log on completion (as it does today) and emit a single progress tick after the HTTP call returns.

### Acceptance criteria

1. While a `design.proteinmpnn` job runs, `~/proteus/logs/<id>.log` grows in real time and contains the underlying script's stdout+stderr.
2. On failure, the log contains the underlying error output (not just the empty file we see today).
3. `jobs.status` for a `design.proteinmpnn` job ticks past `0%` within the first 30 seconds of a real run.
4. Cancelling a running job leaves a partial log with whatever the script had emitted before SIGTERM.
5. `progress=1.0` is reached only after persistence has finished — no regression on `j.Output` carrying the final designs.
6. New tests:
   - `bashRunner` writes interleaved stdout+stderr to a provided `io.Writer` and returns the same content as a string.
   - Each design adapter calls `env.Log.Write` and `env.Progress` at least once during a stubbed `Run`.
   - The job log file is non-empty after a stubbed adapter run.
7. `go test ./...` and `go vet ./...` clean.

### Also covered by Bug 2's fix

`fold.boltz2` and `fold.chai1` share the same `foldJobTool` struct
(`internal/tools/fold/foldjob.go`) and call `t.backend.Run(ctx, t.name, input, log)`
on exactly the same code path. They will suffer empty job logs and
stuck-at-0% progress in the same way. Once `localBackend.Run` and
`modalBackend.Run` stream output and accept a progress callback, those two
fold tools benefit automatically — no additional changes needed beyond
ensuring the `foldJobTool.Execute` closure plumbs `progress` through.

Acceptance criterion 7 (below) should be extended to require a `go test`
that confirms `fold.boltz2`/`fold.chai1` job logs are non-empty after a
stubbed run.

### Out of scope for Bug 2

- **Parsing structured progress from tool output.** Reading `Designing sequence 2/3` lines from `protein_mpnn_run.py` and converting them to a fractional progress is a nice-to-have, not required.
- **Splitting stderr from stdout in the log.** Interleaved is fine — matches how a human running the script would see it.

### Effort estimate

~one day. The `CmdRunner` signature change touches every adapter + installer call site, but it's a mechanical refactor with strong test coverage already on the adapters.

## Bug 3 — The agent cancels long-running design jobs prematurely

### Summary

In the observed run on 2026-05-20, Qwen3.5-35B-A3B-FP8 cancelled a `design.proteinmpnn` job at 2 min 16 s after polling `jobs.status` ten times and never seeing progress move off `0%`. The job's `EstimatedDuration` is `30 * time.Minute` (`internal/tools/design/design.go:60`), so the cancellation was an order of magnitude earlier than the tool itself advertises.

Once Bug 2 is fixed, progress will tick and the agent will likely no longer cancel. But the underlying policy gap — the agent treats a static-progress reading as evidence of hang — is itself worth fixing, because real GPU tools (RFdiffusion, BindCraft) can legitimately run for tens of minutes between visible stages.

### Reproduction

1. Submit any design tool with an absolute path that resolves.
2. Watch the agent poll `jobs.status` repeatedly.
3. When `progress` doesn't change across ~10 polls, the agent invokes `jobs.cancel` and reports failure to the user.

### Root cause

Two contributing factors, neither yet pinned to a single code site:

- **System prompt and tool result formats** do not surface `EstimatedDuration` to the model in a form it can use to decide *how long is too long*. The `jobs.status` tool returns a one-line summary with status + progress; the model has no anchor for what's normal.
- **No grace period.** There's no client-side guard preventing `jobs.cancel` from firing within `EstimatedDuration / k` of a job's start, even if the model decides to call it.

### Proposed fix

Two complementary changes:

1. **Enrich `jobs.status` and `jobs.result` output** to include `elapsed` and `estimated_duration` fields. With this in the response, the model can reason "elapsed is 2m, estimated is 30m, no need to cancel."
   ```
   j_dd3735d5  tool=design.proteinmpnn  status=running  progress=0%  elapsed=2m16s  estimated=30m
   ```

2. **Update `internal/agent/prompts/system.md`** with a short rule: "For long-running tools, treat `progress` < 1.0 with `elapsed < estimated_duration` as healthy. Do not invoke `jobs.cancel` unless the user asks, the job has exceeded `2 × estimated_duration`, or `jobs.result` returns an error."

### Acceptance criteria

1. `jobs.status` / `jobs.result` responses include `elapsed` and `estimated` fields populated from the job's `Started` timestamp and the tool's `EstimatedDuration()`.
2. In a manual repro, the agent does NOT cancel a running `design.proteinmpnn` job within `< estimated_duration` even when progress is `0%`.
3. The system prompt change is unit-tested via a snapshot test against the updated prompt file.
4. **Calibration check (folds in former Bug 8):** the `EstimatedDuration` for each design tool is sanity-checked against a real CPU-mode ProteinMPNN benchmark — 129 aa, `num_designs=3`, `CUDA_VISIBLE_DEVICES=` — measured outside Proteus. If the actual time differs from `EstimatedDuration` by more than 2×, either `EstimatedDuration` is corrected in `internal/tools/design/design.go` or the cancel heuristic in the prompt uses a different ratio. The measurement is recorded in `docs/VERIFICATION.md` as a footnote so future tuning has a baseline.

### Out of scope for Bug 3

- **Forcing the LLM's hand** with a tool-side cancel-budget check. Considered, but better to teach the model than to mechanically block; users sometimes legitimately want to cancel early.

### Effort estimate

~2 hours.

## Bug 4 — Tool errors don't tell the agent what to try differently

### Summary

After Bug 1 caused `target "designs/d_0003.pdb" does not exist`, the agent retried with `./designs/d_0003.pdb` (a no-op variation), then once more, before finally calling `fs.bash` to list the directory and discover the real path. Three round-trips that a single specific error message could have collapsed into one.

This is downstream of Bug 1 (path resolution) but distinct: even after Bug 1 is fixed, the error path for a genuinely missing file should give the agent something to act on.

### Reproduction

Trigger Bug 1's "target … does not exist." Observe the agent retry with cosmetically-different variants of the same relative path before searching the filesystem.

### Proposed fix

When `os.Stat(target)` fails with `ENOENT`, return a richer error that includes:

- The resolved absolute path the tool tried.
- The active workspace root the tool was resolving against.
- A pointer to `fs.read_structure` / `fs.bash` for discovery.

Sketch:

```
design.proteinmpnn: target "designs/d_0003.pdb" not found —
resolved to "/home/gonjim/proteus/projects/default/designs/d_0003.pdb"
(workspace root). Use fs.read_structure or fs.bash to confirm the
file exists, or pass an absolute path.
```

After Bug 1 ships, the resolved path will be correct, and this error becomes purely informative for the case where the user really did fat-finger the filename.

### Acceptance criteria

1. Every design adapter's "target not found" error includes the resolved absolute path and the workspace root.
2. A unit test asserts the substring `resolved to ` appears in the error for each adapter.

### Out of scope for Bug 4

- Other adapter errors (install missing, malformed input). The targeted fix is the path-related one because it's the one we've seen the agent retry-loop on.

### Effort estimate

~1 hour.

## Bug 5 — Status bar shows the launch cwd, not the active project's workspace

### Summary

When the user launches `proteus tui` from any directory, the status bar's third line displays that launch directory (e.g. `/home/gonjim/Projects/proteus` when launched from the source tree). The **actual project workspace** that tools resolve paths against is `$PROTEUS_HOME/projects/default` (today, `/home/gonjim/proteus/projects/default`). The two paths are almost never the same.

This causes the agent (and the user) to reason about relative paths as if they resolved against the displayed directory. They do not.

### Reproduction

1. Launch from the source tree: `cd ~/Projects/proteus && ./bin/proteus tui`.
2. Observe the status bar showing `/home/gonjim/Projects/proteus`.
3. Save a file via `fs.write` with path `notes.md`.
4. Look for it under `/home/gonjim/Projects/proteus/notes.md` — not there.
5. Find it at `/home/gonjim/proteus/projects/default/notes.md`.

### Root cause

`internal/tui/statusbar.go::setProject` and `setContextPercent` are dead today (they appear in `deadcode ./...` because nothing in the production path calls them — see `docs/CODE-REVIEW-2026-05-20.md` §2.2.2). The third status-bar line is being populated by something else, most likely the launch cwd via `os.Getwd()` in the welcome banner (added in commit `ffe4f3b Merge v0.5 TUI mascot welcome banner`). The status bar was never wired to show the actual `workspace` string that `cmd/proteus/main.go:96` computes via `defaultWorkspace()`.

### Proposed fix

1. Make the status bar's third line display the actual `workspace` path, not the launch cwd. Wire it via `statusBarModel.setProject(workspace)` (resurrecting the dead setter — it was clearly there for this) called from `tui.New` with the workspace value already in scope.
2. Truncate cleanly with an ellipsis for narrow terminals (use the existing TUI width-aware helpers).
3. Optionally, on first launch when the user isn't in the workspace, append a small badge like `· cwd != workspace` so power users notice the asymmetry.

### Acceptance criteria

1. Launching `proteus tui` from any directory shows the resolved `$PROTEUS_HOME/projects/default` (or active project) in the status bar.
2. `statusBarModel.setProject` is no longer in `deadcode ./...` output.
3. A TUI snapshot test asserts the status bar contains the workspace path.

### Out of scope for Bug 5

- **Multi-project support** (i.e. listing/switching projects). The status bar today shows whatever single project is active; multi-project work is a v0.6+ feature.

### Effort estimate

~1 hour.

## Bug 6 — Modal backend must also stream log + emit progress (sibling to Bug 2)

### Summary

`modalBackend.Run` (`internal/backends/backend.go:39-44`) has the same end-of-call write to the log as `localBackend.Run` did before Bug 2's fix. Today it calls `b.client.Run(ctx, tool, input)` (an HTTP round-trip to the Modal function), then writes the returned payload to the log on completion. There is no progress tick — Modal jobs sit at `0%` until they finish.

Bug 2's signature changes already pull this code path along (`Backend.Run` gains the `progress` parameter), but the Modal-specific behavior — emit at least one `progress` tick after the HTTP call succeeds, and write a single line to the log to indicate the call has started — needs its own implementation and test.

### Proposed fix

Inside `modalBackend.Run`, after Bug 2's interface change:

```go
func (b *modalBackend) Run(ctx context.Context, tool string, input []byte, log io.Writer, progress func(float64)) ([]byte, error) {
    fmt.Fprintf(log, "modal: dispatching %s\n", tool)
    progress(0.05)
    out, err := b.client.Run(ctx, tool, input)
    if err != nil {
        fmt.Fprintf(log, "modal: %s failed: %v\n", tool, err)
        return nil, err
    }
    progress(0.9)
    _, _ = log.Write([]byte(out))
    return []byte(out), err
}
```

### Acceptance criteria

1. A Modal `fold.boltz2` job's log shows the `modal: dispatching` line within a second of submission.
2. The job's `progress` ticks to `0.05` after the dispatch and to `0.9` on completion.
3. A new test in `internal/backends/modal/client_test.go` exercises this with a stub `Client`.

### Out of scope for Bug 6

- **Real-time progress from inside Modal functions.** That would require a callback channel (websocket, polling, …) and is a v0.7 topic. A single tick is enough to signal "alive" to the agent.

### Effort estimate

~1 hour. Most of the change is mechanical once Bug 2 lands.

## Bug 7 — Cancellation leaves orphan child processes

### Summary

`Manager.Cancel` cancels the job's context, which propagates to the `exec.CommandContext`-wrapped `bash -c` invocation. `exec.CommandContext` will SIGKILL the bash process, but the **bash subprocess group** (the `python protein_mpnn_run.py …` child) doesn't necessarily exit unless the kill propagates through. On Linux, the default behavior is that signals go to the parent only; the children inherit and keep running, attached to PID 1.

This is suspected — not confirmed — from the j_dd3735d5 cancellation. A clean repro requires watching `ps -ef` for orphan `python` processes after a cancel.

### Reproduction (to be confirmed)

1. Submit `design.proteinmpnn` so it actually starts `protein_mpnn_run.py`.
2. Wait until the python process is visible in `ps -ef | grep protein_mpnn_run`.
3. From the agent or directly, call `jobs.cancel`.
4. Inside ~5 s, the python process should be gone. If `ps` still shows it (reparented to PID 1), this bug is confirmed.

### Proposed fix

Make `bashRunner` create a new process group for its child and propagate cancellation across the whole group:

```go
import "syscall"

func bashRunner(ctx context.Context, dir, command string, log io.Writer) (string, error) {
    cmd := exec.CommandContext(ctx, "bash", "-c", command)
    if dir != "" { cmd.Dir = dir }
    cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true}
    cmd.Cancel = func() error {
        // Kill the whole process group, not just bash.
        return syscall.Kill(-cmd.Process.Pid, syscall.SIGTERM)
    }
    cmd.WaitDelay = 5 * time.Second // SIGKILL after grace period
    var buf bytes.Buffer
    cmd.Stdout = io.MultiWriter(&buf, log)
    cmd.Stderr = io.MultiWriter(&buf, log)
    err := cmd.Run()
    return buf.String(), err
}
```

The `Setpgid` makes the bash process the leader of a new process group; `syscall.Kill(-pid, SIGTERM)` then sends SIGTERM to every process in that group. `WaitDelay` ensures a SIGKILL follows if the grace period expires.

### Acceptance criteria

1. New test: spawn `bash -c "sleep 30 & wait"`, cancel via context, then check that the `sleep` process has terminated within 6 s.
2. Manual repro of the original scenario: cancelled `design.proteinmpnn` leaves no orphan `python` process in `ps`.

### Out of scope for Bug 7

- **Windows / macOS process-group behavior.** Proteus targets Linux on GB10 / DGX; Windows isn't supported. macOS support is best-effort. The syscall portability is acceptable.

### Effort estimate

~2 hours, dominated by the test.

### Sequencing note

Bug 7 touches `bashRunner` — the same function Bug 2 modifies. They must be done **sequentially**: implement Bug 2 first (signature change + stdout/stderr tee), then layer Bug 7's process-group logic on top, OR implement them as a single PR.
