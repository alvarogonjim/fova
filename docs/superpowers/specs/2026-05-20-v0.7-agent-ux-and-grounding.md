# v0.7 — Agent UX and grounding

**Date opened:** 2026-05-20
**Last updated:** 2026-05-20 (extended with workstreams B, C, D)
**Status:** Open — scoped, ready to dispatch
**Component:** `internal/tui/` (slash menu + commands), `internal/agent/prompts/`, `internal/tools/{knowledge,lab,plan}/`, `internal/backends/local/` (container backend), `internal/transport/` (new)
**Affects since:** v0.4 (slash menu / system prompt), v0.3 (corpus), v0.3 (plan.create), v0.1 (local backend exec model)

## Why this ticket exists

Manual end-to-end testing of v0.6 on the GB10 surfaced a tangle of issues that
share a common shape: **the agent's view of the world and the user's view of
the world drift apart**, and **the tools the agent reaches for don't actually
run on this GPU**. The agent invents slash commands the TUI doesn't have,
guesses tool names from a different naming convention, runs a multi-minute
corpus map with no visible progress, fabricates citation evidence that doesn't
trace back to the corpus, and the slash menu itself doesn't help the user
discover real sub-commands. Underneath all that, the local backend's per-tool
uv venv model can't carry sm_121-capable PyTorch wheels for the GB10, so even
when the agent calls the right tool with the right arguments, the tool fails
on the GPU.

v0.7 addresses the whole stack in four workstreams:

- **A. Agent UX and grounding (Bugs 1-7)** — fixes from the original v0.6
  testing session: slash menu, system prompt, corpus surface, render quality.
- **B. Search and planning hardening (Bugs 9-12)** — robustness pass on
  corpus.search, lab.targets_search, and plan.create beyond the citation
  validation already in Bug 5.
- **C. GB10 GPU via containers (Bug 13)** — replace the per-tool uv venv with
  a per-tool NGC PyTorch container. Detect Podman or Docker at install time.
  Bug 8 (bindcraft `requirements.txt`) is folded into the bindcraft
  Containerfile under workstream D.
- **D. All-tool installer audit (Bugs 14-22)** — one Containerfile + smoke
  test per tool in `tools.toml` (ipsae, proteinmpnn, rfdiffusion, bindcraft,
  ligandmpnn, rfantibody, rfdiffusion2, boltz2, chai1). Final acceptance is
  manual GB10 verification by the maintainer after merge.

Bug 1 and Bug 2 are user-prioritised (the direct asks from the v0.6 debug
session). Bugs 3, 4, 5 came out of the same session and belong with them. The
later bugs are scoped from the v0.7 brainstorm (see `[v0.7 design](.)` —
this document is its own design record).

This document follows the same structure as the v0.6 ticket
(`2026-05-20-v0.6-design-path-resolution.md`): numbered bugs, each with
Summary, Reproduction, Root cause, Proposed fix, Acceptance criteria,
Out-of-scope, Effort. Bug 13 is architectural; it deviates slightly to
document the runtime, base image, and migration story.

## Bug 1 — Slash menu doesn't show sub-commands or argument hints

### Summary

Typing `/plan` and pressing Enter opens the plan view, but typing `/plan ` (with a trailing space) or `/plan ap…` does NOT show `/plan approve` / `/plan cancel` in the autocomplete popup. The same gap exists for `/install` — typing `/install ` should offer the list of installable tool names from `tools.toml` (`bindcraft`, `proteinmpnn`, `rfdiffusion`, …). And for `/model`, `/auth`, etc. The user has to already know the sub-commands and arguments to use them. There is no in-TUI way to discover them.

This is the user's explicit ask: *"when I type `/plan` it does not show the options"* and *"include the help with the /install command too and other commands."*

### Reproduction

1. Launch `proteus tui`.
2. Type `/plan` in the input. The slash menu shows the single `/plan` entry.
3. Type a space → `/plan `. The slash menu disappears.
4. Type `/plan ap` → no autocomplete.
5. The only way to learn that `/plan approve` exists is to read the source or have the agent mention it (and the agent itself sometimes invents non-existent variants — see Bug 2).

### Root cause

`internal/tui/commands.go` declares a flat catalogue of top-level commands. The matcher in `internal/tui/slashmenu.go` filters that catalogue by prefix. Neither knows about per-command sub-arguments. The `/plan approve` / `/plan cancel` handling lives inline in `internal/tui/app.go::handleSlashCommand` (or equivalent), with no surface that the slash menu can query.

### Proposed fix

1. **Extend the command catalogue schema** in `internal/tui/commands.go` with two optional fields per entry:
   ```go
   type Subcommand struct {
       Name        string // e.g. "approve"
       Description string // e.g. "Approve the current design plan"
   }

   type ArgumentSource int
   const (
       ArgsNone ArgumentSource = iota
       ArgsInstalledTools         // for /install, /doctor: list from tools.toml
       ArgsModels                 // for /model: list from the model registry
       ArgsAuthProviders          // for /auth: list known provider names
       ArgsFreeText               // free-text argument; show usage hint only
   )

   type Command struct {
       Name        string
       Description string
       Subcommands []Subcommand   // keyword sub-commands (/plan approve)
       Arguments   ArgumentSource // dynamic argument completion
       Usage       string         // shown as a hint when Arguments == ArgsFreeText
   }
   ```
2. **Populate sub-commands and argument sources for every command.** Audit `app.go::handleSlashCommand` (or wherever slash command routing lives) to find every command that today branches on a second word. The known set:
   - `/plan` → `Subcommands: [approve, cancel]`
   - `/install` → `Arguments: ArgsInstalledTools`
   - `/uninstall` → `Arguments: ArgsInstalledTools`
   - `/model` → `Arguments: ArgsModels`
   - `/auth` → `Arguments: ArgsAuthProviders`
   - `/doctor`, `/keys`, `/theme`, `/reload`, `/replay`, … → `Arguments: ArgsNone`
3. **Teach the slash menu to surface them.** When the input is `/<cmd> ` (with trailing space) or `/<cmd> <prefix>`:
   - If the command has `Subcommands`, show matching sub-command rows.
   - If it has a dynamic `Arguments` source, fetch the live list (e.g. for `/install`, read `tools.toml`'s `[tools.*]` keys) and show matching rows.
   - If `Arguments == ArgsFreeText`, show a one-line usage hint instead of a list.
   - Navigation (`↑/↓`, `Tab`, `Esc`) works the same as for top-level commands.
4. **Bare `/<cmd>`** (no space) continues to behave as today — the top-level command. Adding a trailing space is the trigger for help.
5. **Dynamic argument sources resolve at suggest time, not at startup**, so the install list reflects newly-added tools and the model list reflects `/model`-driven changes.

### Acceptance criteria

1. Typing `/plan ` opens a popup listing `/plan approve` and `/plan cancel` with descriptions.
2. Typing `/plan ap` narrows to `/plan approve` only.
3. Typing `/install ` opens a popup listing every tool name in `tools.toml` (`bindcraft`, `proteinmpnn`, `rfdiffusion`, `rfantibody`, `rfdiffusion2`, `ligandmpnn`, `chai2`, `boltz2`, `chai1`, `ipsae`).
4. Typing `/install pr` narrows to `proteinmpnn` only.
5. Typing `/model ` opens a popup listing the registered model IDs from the active config.
6. `Tab` completes a sub-command or argument into the input.
7. `Esc` dismisses without committing.
8. Snapshot tests of the slash menu view contain the sub-command rows for `/plan` and the argument rows for `/install`.
9. The existing top-level slash-menu behavior is unchanged.

### Out of scope

- **Sub-sub-commands.** No command in v0.7 has them. If they appear later, extend the schema then.
- **Completing PDB IDs, file paths, or other free-form arguments.** Only keyword sub-commands and registry-derived lists (tools, models, providers) are in scope; arbitrary string completion is v0.8+.

### Effort estimate

~half a day. Catalogue extension, matcher extension, three suggesters (tools/models/auth), snapshot tests.

## Bug 2 — The agent invents slash commands the TUI doesn't have

### Summary

After `plan.create`, Qwen3.5-35B-A3B-FP8 told the user: *"Reply `/approve` to proceed or `/revise` with modifications."* Neither command exists; the user typed both and got `unknown command`. The real commands are `/plan approve` and (for "revise") just describing the change in chat. This is the user's other explicit ask: *"I would like that the chat tells me real commands."*

### Reproduction

In a fresh session, ask the agent to plan a design for any target. After `plan.create` lands, the agent's natural-language follow-up usually invents a slash command (the exact verb varies across runs — `/approve`, `/revise`, `/edit`, `/yes`, …).

### Root cause

The system prompt at `internal/agent/prompts/system.md` doesn't tell the model what slash commands exist. The model improvises plausible-looking ones from prior training. There's no grounding loop — the model never gets feedback that `/approve` isn't real (because `unknown command` is rendered to the user, not back into the chat history the model sees).

### Proposed fix

Two complementary changes, both modest:

1. **Embed the slash-command catalogue into the system prompt** at load time. `internal/agent/prompts.go` (the loader) reads `system.md` and templates in the actual catalogue from `internal/tui/commands.go` so the model has the ground truth on every turn. The injected block reads (illustrative):
   ```
   The user interface offers these slash commands (no others exist):
     /plan         — view current design plan
     /plan approve — approve and commit it
     /plan cancel  — discard it
     /model <id>   — switch LLM
     /doctor       — check tool install status
     …
   When suggesting next steps, refer to these commands literally. Never
   invent a slash command. If a needed verb doesn't exist as a command,
   tell the user to describe the change in plain English instead.
   ```
2. **Test the grounding holds.** A snapshot test asserts the rendered system prompt contains every command name returned by the catalogue.

### Acceptance criteria

1. The system prompt, as actually sent to the LLM, lists every slash command (top-level and sub-command from Bug 1) verbatim.
2. A snapshot test asserts the prompt contains a known set of commands (`/plan`, `/plan approve`, `/plan cancel`, `/model`, `/doctor`, etc.).
3. In manual repro after `plan.create`, the agent's follow-up message refers to `/plan approve` rather than `/approve`. (This is best-effort — LLMs hallucinate; the prompt makes it materially less likely.)

### Out of scope

- **Validating the agent's text post-hoc** to catch invented commands. That's a much bigger lift (a guard that scans assistant messages for `/[a-z]+` and validates against the catalogue). Defer.
- **Auto-correcting** user mistypes. Different problem.

### Effort estimate

~2 hours. The grounding block is small; the snapshot test guarantees it stays accurate as the catalogue grows.

### Dependency note

Bug 2 is best landed *after* Bug 1: the prompt will then include the new sub-commands, so the agent learns about `/plan approve` and stops inventing `/approve`. If Bug 2 ships first, the prompt will list the top-level commands without sub-arguments and the agent will still guess.

## Bug 3 — Dotted sub-tool names that don't exist

### Summary

The agent guessed `knowledge.corpus.add` (the natural OpenAI-style dotted name) instead of the actual `knowledge.corpus` umbrella tool with an `action="add"` field. The unknown-tool error fired, and the agent retried with the correct name — costing one extra round-trip. Same pattern will recur for every umbrella tool in the registry.

### Reproduction

In a fresh session: "Add this paper to the corpus: …". With probability >50 % Qwen3.5 calls `knowledge.corpus.add`. The retry shows up as a `✗ unknown tool` line in the chat trace.

### Root cause

Two competing tool-naming conventions live in the registry simultaneously:

- The **flat namespaced style** used everywhere else: `design.proteinmpnn`, `fold.esmfold`, `score.ipsae`, `lab.targets_search`, …
- The **umbrella + action style** used by `knowledge.corpus`: one tool, eight actions selected via a field on the JSON input.

LLMs trained on OpenAI tool-calling default to the flat style. The umbrella is an outlier.

### Proposed fix

Two options; pick one:

**Option A — Flatten the umbrella into eight tools (preferred).** Register `knowledge.corpus_add`, `knowledge.corpus_search`, `knowledge.corpus_grep`, `knowledge.corpus_map`, `knowledge.corpus_reduce`, `knowledge.corpus_read`, `knowledge.corpus_remove`, `knowledge.corpus_add_from_search` as eight separate tools, each forwarding to the existing dispatcher. The internal `corpus.go` dispatcher stays — it now has eight callers instead of one.

(Note: `corpus.add` would collide with Go field naming if we used a dot; use underscore as the action separator in tool names. Or use dots if the registry already permits multi-dot names — verify.)

**Option B — Enrich the unknown-tool error.** When the registry sees an unknown tool name that *starts with* a known umbrella tool's name, return an error that lists the umbrella's allowed actions. e.g.:
> `unknown tool "knowledge.corpus.add" — did you mean knowledge.corpus with action="add"? Available actions: add, search, grep, map, reduce, read, remove, add_from_search`

This is generalised Bug 4 from the v0.6 ticket: errors that help the agent self-correct.

Option A is the cleaner long-term fix; Option B is faster and benefits other umbrella tools if any appear later. **Pick A.**

### Acceptance criteria

1. The registry exposes eight `knowledge.corpus_*` tools (or `knowledge.corpus.*` if dots are valid in tool names — pick one style and document it).
2. The original `knowledge.corpus` umbrella is removed (or deprecated and dispatches to the new tools internally for one release).
3. Existing corpus tests pass against the new tools.
4. A manual repro shows the agent calling `knowledge.corpus_add` directly without retrying.

### Out of scope

- Auditing every other tool for naming consistency. No other umbrella exists today.

### Effort estimate

~half a day. The dispatcher is already there; the work is constructing eight tool wrappers + updating tests.

## Bug 4 — `knowledge.corpus map` runs for minutes with no progress signal

### Summary

After `corpus.map` was invoked, the TUI showed *"Searching… (93s · esc to interrupt)"* and kept ticking with no indication of what was happening. In this session the operation took ~2-3 minutes (30 papers × LLM-bounded fan-out on Qwen3.5-35B). During that window the user has no way to distinguish "working" from "hung" without polling `nvidia-smi`. The fix v0.6 Bug 2 made for async jobs (design.*, fold.boltz2/chai1) does NOT apply to synchronous tools like `knowledge.corpus`, which run in-process from the agent loop.

### Reproduction

Ask the agent to "plan a design for any target with literature evidence." The plan-from-target skill calls `corpus.add` then `corpus.map`. The map step blocks the agent loop with no per-paper feedback.

### Root cause

`internal/tools/knowledge/corpus.go::cmdMap` uses a bounded-concurrency goroutine fan-out (semaphore + WaitGroup, well-engineered for correctness) but emits no progress events. There's no `progress` callback in the corpus tool's `Execute` signature, and no message-bus integration for sync tools to surface "mapping paper 7/30…" notes to the chat.

### Proposed fix

Option A (preferred) — **make corpus.map an async job**: take the same path that `design.*` and `fold.boltz2/chai1` already take. The tool's `Execute` returns a `JobID`, the agent polls via `jobs.status` (which now carries `elapsed`/`estimated` from v0.6 Bug 3), and the fan-out emits progress ticks. `corpus.add` and the other cheap actions stay synchronous.

Option B — **synchronous with a chat-side progress channel**: pass an `events chan<- string` into the corpus tool from the agent loop; the loop renders each event as a `⎿ corpus.map: 7/30 papers` line in the chat. This is the "TUI-only" progress surface and doesn't touch the jobs system.

Option A is the consistent architectural choice (corpus.map IS a long job, just like a design tool); Option B is cheaper but creates a new progress mechanism. **Pick A** unless implementation reveals corpus.map can't tolerate the jobs.Spec model (it should — it's just LLM calls).

### Acceptance criteria

1. `knowledge.corpus_map` (or `knowledge.corpus action=map`) returns a `JobID` and the actual map runs in the background.
2. `jobs.status` for the map ticks per-paper progress (e.g. `progress=0.23  elapsed=42s  estimated=2m0s` after 7/30 papers).
3. Cancelling the map job cleanly terminates outstanding paper-level LLM calls.
4. The `plan-from-target.md` skill is updated to instruct the agent to poll the job (matches v0.6 Bug 3's polling rule).

### Out of scope

- **Streaming partial map results back to the chat as each paper finishes.** Nice; not required. The job-result-on-completion model is enough.

### Effort estimate

~half a day. Mostly mirrors v0.6 Bug 2's plumbing pattern.

## Bug 5 — `plan.create` accepts evidence that doesn't trace back to the corpus

### Summary

The plan emitted in this session listed *"[Dunbrack et al. 2024] — BindCraft achieves high success rates on lysozyme with ipSAE > 0.50 and Kd < 1 μM"* as evidence. Dunbrack is a real researcher but the BindCraft paper is by **Pacesa et al. 2024** (Baker lab) — not by Dunbrack. The corpus.map step (the slow one we just waited through) was supposed to surface real DOIs from the 30 Europe PMC papers it processed; the citation in the plan didn't come from the corpus, it was hallucinated by the LLM as it constructed the `plan.create` arguments.

`plan.create` accepts the evidence field as free-form text and doesn't validate it against the project's corpus.

### Reproduction

Ask the agent to plan a design for any target with literature evidence. Inspect the resulting `DesignPlan.Evidence` list; compare each citation against the `corpus_papers` table. With high probability at least one entry doesn't have a matching `paper_id`.

### Root cause

`internal/tools/plan/plan.go::PlanCreate` (or whatever the tool's named) accepts evidence entries as strings or as `{title, doi, …}` structs without cross-referencing the corpus store. There's no validation step.

### Proposed fix

1. **Change the `plan.create` schema** to require each evidence entry to carry a `corpus_paper_id` field — the ID returned by `corpus.add` / `corpus.search`. Free-text-only entries are rejected.
2. **Validate inside `plan.create`** that each `corpus_paper_id` exists in the active project's corpus. Unknown IDs cause the tool to fail with an actionable error.
3. **`plan.create` formats the citation** itself from the corpus paper's stored metadata (authors, year, title, DOI) — the agent doesn't get to free-write the citation string at all.
4. **Update `plan-from-target.md`** to make this contract explicit: "After `corpus.map`, when calling `plan.create`, pass the `corpus_paper_id` of each evidence paper. Do not write citation strings yourself."

### Acceptance criteria

1. `plan.create` rejects evidence entries that lack a `corpus_paper_id`.
2. `plan.create` rejects entries whose `corpus_paper_id` isn't in the corpus.
3. The rendered evidence list in `/plan` shows citations derived from corpus metadata, not LLM-authored text.
4. New test in `internal/tools/plan/plan_test.go` covers both rejection paths.
5. The `plan-from-target.md` skill text is updated.

### Out of scope

- **Cross-referencing free-text claims inside the rationale.** The rationale text itself is still LLM-authored; that's harder to ground and isn't this bug.
- **Auto-citing external sources beyond the corpus.** Evidence comes from the corpus, full stop.

### Effort estimate

~half a day. Schema change + validation + rendering + skill text.

## Bug 6 — `/plan` view renders as a single squashed paragraph

### Summary

`/plan` displays the active `DesignPlan` as one long unbroken paragraph: target, application, method, filters, shortlist size, compute, cost, time, rationale, and evidence all run together with no line breaks. By contrast, the table emitted by `plan.create` (when the plan is first created) renders cleanly as a labelled table — the same data, but readable. The view rendering doesn't match the creation rendering.

Concrete example from this session (the actual output, unedited):

> Design plan p_19b6e95a-c47a-427a-aefb-8d9e2bf106bf — approved (2026-05-20 12:45 UTC) Target: 1LYZ chain A Application: binder Method: BindCraft (fallback: RFdiffusion+ProteinMPNN) Filters: (none set) Shortlist: 50 Compute: default Estimate: $15.00 USD · 45 minutes Rationale: …

### Root cause

The `/plan` view handler in `internal/tui/app.go` (or a sibling file) builds the plan text as a single string with no newlines (or with `\n` that the chat renderer collapses to spaces). Compare this to `internal/tools/plan/plan.go`'s `Result.Display` which builds a structured table with explicit row separators — *that* one renders correctly because the chat renderer preserves its format.

### Proposed fix

1. Have the `/plan` view handler emit the **same labelled-table format** as `plan.create`'s `Display`. Extract the table-rendering helper into a shared function (e.g. `internal/tools/plan/render.go::RenderPlan(*domain.DesignPlan) string`) and call it from both surfaces.
2. Verify the chat renderer preserves the table's line structure. If it doesn't, fix the renderer (probably in `internal/tui/chat.go`) — newlines inside slash-command output must be honoured.
3. Replace the trailing-line instructions with the actual sub-commands (depends on Bug 1's schema):
   > `Use /plan approve to lock it in, or /plan cancel to discard it.`
   becomes prompt text that lives in the catalogue, not free-form Display text.

### Acceptance criteria

1. `/plan` for an approved plan renders the same labelled-table format as the one emitted by `plan.create`.
2. Long fields (rationale, evidence list) wrap at the panel width, with each evidence entry on its own indented line.
3. A snapshot test renders a known plan and asserts the output contains `Target:`, `Application:`, `Filters:` on separate lines.

### Out of scope

- Re-styling the table with colours / boxes / fancy ASCII art. Plain labelled rows are enough; matching the existing `plan.create` output is the bar.

### Effort estimate

~2 hours. Mostly the extraction of the shared renderer.

## Bug 7 — `/doctor` output renders as a single squashed paragraph

### Summary

Same root cause as Bug 6, different command. `/doctor`'s output today looks like:

> System ok  uv: /home/gonjim/.local/bin/uv  Local protein tools --  bindcraft      not installed (run: /install bindcraft) --  boltz2         not installed (run: /install boltz2) --  chai1          not installed (run: /install chai1) ok  ipsae          1.0.0 --  ligandmpnn     …

Every tool's status runs into the next. Expected: one tool per line, aligned columns, easy to scan.

### Root cause

`internal/tui/setup.go::cmdDoctor` (or wherever `/doctor` posts its report) builds the report as a single string without preserving line breaks the chat renderer respects. The data is already structured (`doctor.Report`) — the renderer just throws away the structure.

### Proposed fix

1. Emit one line per tool:
   ```
   System
     ok  uv: /home/gonjim/.local/bin/uv

   Local protein tools
     ok    ipsae          1.0.0
     ok    proteinmpnn    1.0.1
     ──    bindcraft      not installed   /install bindcraft
     ──    boltz2         not installed   /install boltz2
     ──    chai1          not installed   /install chai1
     …
   ```
2. Reuse the table-renderer infrastructure from Bug 6 if extracted.
3. A status-marker convention (`ok`, `──`, `!!`, `??`) is consistent across `/doctor` rows.

### Acceptance criteria

1. `/doctor` output has one tool status per line.
2. Status markers align in a column.
3. Tool names align in a column.
4. The `/install <tool>` hint sits in its own column on the right.
5. A snapshot test asserts the output contains at least 3 newlines between `System` and the last tool row.

### Out of scope

- Adding new health checks beyond what `doctor.Report` already produces.

### Effort estimate

~1 hour. Pure render fix.

### Sequencing note

Bugs 6 and 7 share the same renderer infrastructure question (does the chat preserve newlines in slash-command output?). Fix the renderer once, apply to both commands.

## Bug 8 — `tools.bindcraft` install recipe references a `requirements.txt` that doesn't exist upstream

> **Status: Superseded by Bug 14 (bindcraft Containerfile).** The container
> rewrite (Bug 13) replaces the venv-based `install_steps` recipe entirely.
> The PyRosetta-at-build-time approach noted below is preserved as part of
> Bug 14's Containerfile. The content below is retained for context.

### Summary

`/install bindcraft` fails on Step 4 with `error: File not found: requirements.txt`. The recipe in `internal/backends/local/tools.toml` runs `uv pip install -r requirements.txt`, but the [martinpacesa/BindCraft repo](https://github.com/martinpacesa/BindCraft) has no `requirements.txt` — it ships an `install_bindcraft.sh` conda script instead. The Proteus recipe references a file that has never existed in upstream.

Surfaced this session as `j_43cd5953-d98d-45b7-a61b-a91cbfb50cbd`; 58 s into the install (after the ~3 GB torch download finished), then the recipe blew up.

### Reproduction

```
/install bindcraft
```

Watch `~/proteus/logs/<id>.log` — torch downloads, then the requirements.txt step fails.

### Root cause

`internal/backends/local/tools.toml::[tools.bindcraft]::install_steps[3]` is:

```toml
"uv pip install --python {{ venv_dir }}/bin/python -r requirements.txt",
```

There is no such file in the cloned tree (`/home/gonjim/proteus/tools/bindcraft/` after `git clone …`).

### Proposed fix

Replace Step 4 with an explicit list of the Python packages BindCraft actually imports. Derive the list from the upstream `install_bindcraft.sh` conda spec (the source of truth for BindCraft's deps), translating conda package names to pip equivalents. The list (as of upstream commit at time of writing — re-verify before landing):

```toml
install_steps = [
  "git clone {{ repo }} .",
  "uv venv --python {{ python }} {{ venv_dir }}",
  "uv pip install --python {{ venv_dir }}/bin/python --torch-backend=auto torch==2.8.0",
  # was: -r requirements.txt — file doesn't exist upstream.
  # The deps come from install_bindcraft.sh in the upstream repo.
  "uv pip install --python {{ venv_dir }}/bin/python \
     biopython numpy pandas matplotlib scipy scikit-learn \
     pdbfixer openbabel jax flax dm-haiku ml-collections optax \
     immutabledict py3Dmol seaborn",
  "uv pip install --python {{ venv_dir }}/bin/python git+https://github.com/sokrypton/ColabDesign.git",
  "uv pip install --python {{ venv_dir }}/bin/python pyrosetta-installer",
  "{{ venv_dir }}/bin/python -c 'import pyrosetta_installer; pyrosetta_installer.install_pyrosetta(serialization=True)'",
]
```

**Action item for the implementer:** before committing this list, run `install_bindcraft.sh` in a throw-away conda env and verify what actually gets installed; then trim the pip list to the minimum that imports cleanly into a fresh venv. Some packages (e.g. `openbabel`, `pdbfixer`) historically have only-on-conda problems; we may need to source them from a non-default index, or drop them and let BindCraft skip the optional features they enable.

### Acceptance criteria

1. `/install bindcraft` completes successfully on a clean checkout (network + ~10 GB of space).
2. `/doctor` reports `bindcraft` as `installed`.
3. `python -c "import bindcraft"` (or whatever the upstream entry point is) succeeds in the new venv.
4. The recipe carries a comment pointing at the upstream `install_bindcraft.sh` so future drift is easy to diagnose.

### Out of scope

- **Other broken install recipes.** Audit other tools (`rfantibody`, `rfdiffusion2`, `ligandmpnn`, `chai2`, `boltz2`, `chai1`) under a follow-up. They likely have similar drift since none were verified end-to-end on the GB10 (per `docs/VERIFICATION.md` v0.4 #5).

### Effort estimate

~half a day, dominated by the manual verification step (running the install and confirming all the imports work).

### Related — GPU still blocks the actual run

Even with the install recipe fixed, BindCraft requires `sm_121`-capable PyTorch to actually run on the GB10. This ticket fixes the *install*; running on GB10 still depends on the NGC PyTorch path mentioned in `docs/VERIFICATION.md` AC notes.

# Workstream B — Search and planning hardening

## Bug 9 — `corpus.search` returns empty or low-signal results without telling the agent

### Summary

`corpus.search` against Europe PMC sometimes returns `[]` (the query missed),
sometimes returns 50+ results dominated by reviews and unrelated work, and
sometimes returns a partial list when PMC is rate-limiting or returning 5xx.
In every case the tool's return signature is the same `{papers: [...]}`, so
the agent can't distinguish "the corpus really doesn't have this" from "PMC
hiccuped — retry" from "your query was too narrow". The agent then either
gives up or fabricates citations from training data (this is the upstream
cause of Bug 5; tightening citation validation alone doesn't solve the
discovery problem).

### Reproduction

1. Ask the agent for literature on a deliberately narrow target ("ipSAE
   scoring on de novo TIM-barrel binders"). PMC returns `[]`; the agent
   reports "no relevant papers" without trying to broaden.
2. Re-run the same query 10x while throttled: ~1-2 runs return partial
   results because PMC 503'd partway through pagination.

### Root cause

`internal/tools/knowledge/corpus.go::cmdSearch` calls Europe PMC with no
retry, no timeout, no result-shape signalling, and no query refinement
suggestion. Empty results are returned bare; 5xx surfaces as a generic error.

### Proposed fix

1. **Structured empty contract.** Return `{papers: [], note: "no results — try
   broader terms or fewer filters", query_sent: "<sanitised>"}` when PMC
   returns zero hits. The agent sees the note in the tool output and
   self-corrects without fabricating.
2. **Retry policy via the new shared wrapper.** All PMC calls go through
   `internal/transport/http.go` (introduced as part of Bug 12). 3 attempts
   with exponential backoff (1s, 3s, 9s) on 5xx and connection timeouts;
   permanent failure on 4xx.
3. **Query sanitisation.** Strip stop-tokens (`the`, `a`, `of`, …), trim,
   collapse internal whitespace, and quote multi-word literal terms. Log the
   actual sent query in the tool output's `query_sent` field so debug
   sessions can verify what PMC saw.
4. **Junk filter.** Drop results without a title or abstract. Cap at 50
   results; if PMC returned more, set `truncated_at: 50` in the response.
5. **Test fixtures.** Three table-driven cases: 5xx-then-200 verifies retry,
   empty 200 verifies structured-empty, 50+ result body verifies truncation.

### Acceptance criteria

1. `corpus.search` against an unknown query returns `papers=[]` with a
   `note` field; a unit test asserts the field is present.
2. A fixture-driven test simulates a 503 followed by a 200 and asserts the
   final result is the 200 body (retry worked).
3. Sanitised query appears in `query_sent` and is what the test fixture asserts
   PMC received.
4. Manual repro: the narrow-target session in Reproduction no longer
   fabricates citations because the agent has a "broaden the query" hint.

### Out of scope

- **Cross-source aggregation.** v0.7 stays on Europe PMC; bioRxiv / Semantic
  Scholar / OpenAlex are v0.8+.
- **Semantic re-ranking** of results inside `corpus.search`. The existing
  corpus.map step handles relevance; search just returns hits.

### Effort estimate

~half a day. The retry wrapper is shared (Bug 12), so this is mostly
contract changes + tests.

## Bug 10 — `lab.targets_search` is brittle: silently mis-returns chain "A", no retry, no fallback

### Summary

`lab.targets_search` (PDB / UniProt resolver) has three quiet failure modes:
(a) when the PDB entry has no `chain` info, the code falls back to chain "A"
without saying so — the agent then runs design on the wrong chain; (b) PDB
503s are unhandled — the tool returns "no entry found" and the agent gives
up on a target that actually exists; (c) when PDB doesn't have a structure
for a target the user asked for by gene name, there's no UniProt fallback —
the tool reports failure instead of offering "structure unknown; here's the
UniProt sequence + closest structural homolog".

### Reproduction

1. Search for `1LYZ` → returns chain `A`. Correct, but only because of the
   silent fallback — try a PDB entry with no chain list (e.g. an obsolete
   entry) and the fallback still picks `A`.
2. Search during a PDB outage → "not found", even though the entry exists.
3. Search for an enzyme by gene name with no PDB entry → empty result
   instead of UniProt fallback.

### Root cause

`internal/tools/lab/tools.go` (the targets_search command) calls the PDB
REST API with one attempt, parses the chain list optimistically, and never
falls through to UniProt. Cache hits don't exist — every query is a
round-trip.

### Proposed fix

1. **Explicit chain handling.** If the PDB entry returns no chain list, the
   tool surfaces `chain: null` and `chain_inference_note: "PDB entry has no
   chain metadata; specify a chain explicitly"`. The agent must pass an
   explicit chain to design tools or the design tool refuses.
2. **Retry via shared wrapper (Bug 12).** Same policy as Bug 9 — 3 attempts
   on 5xx with exponential backoff.
3. **UniProt fallback.** If the PDB query returns no hit *and* the input
   looks like a gene/protein name, query UniProt; return the canonical
   sequence and the closest experimental structure (linked PDB ID via
   UniProt's PDB cross-reference, if any).
4. **Local cache.** `~/.proteus/cache/targets.db` (BoltDB; one bucket
   `targets`, key=PDB-ID or UniProt-AC, value=JSON, TTL=7 days). Cache hits
   are noted in the tool output (`cached: true`) so debug sessions can tell
   stale vs fresh.
5. **Tests.** Fixtures for: 1LYZ chain A round-trip (happy path), missing
   chain metadata, PDB 503 retry, UniProt fallback when PDB misses.

### Acceptance criteria

1. A search against an obsolete PDB entry with no chain list returns
   `chain: null` (no silent "A" fallback).
2. A test fixture replays PDB 503 followed by 200; the tool returns the
   200 body.
3. A search for a gene name with no PDB structure returns a UniProt
   record with sequence + closest-structure cross-ref, not "no results".
4. Two consecutive identical searches: the second is served from
   `~/.proteus/cache/targets.db` and emits `cached: true`. The TTL is
   honoured (write a cache entry with a forced past timestamp, confirm it
   re-fetches).

### Out of scope

- **AlphaFoldDB fallback** for cases where no experimental structure
  exists. Worth it but v0.8+. Document in the tool description that
  unstructured targets need explicit user direction for now.
- **Multi-organism disambiguation** for ambiguous gene names. UniProt
  returns the reviewed/canonical entry; v0.8+ can surface alternatives.

### Effort estimate

~half a day. UniProt client is new but small; BoltDB cache is one file.

## Bug 11 — `plan.create` accepts plans the system can't actually execute

### Summary

`plan.create` today validates `corpus_paper_id` (after Bug 5) but does no
other cross-checks. The user can land a plan with `method = "BindCraft"`
when bindcraft isn't installed, `application = "enzyme_design"` paired with
`method = "BindCraft"` (BindCraft is binder-only), `ipsae > 1.0` (impossible
— ipSAE is in [0, 1]), `shortlist = 5000` (exceeds any reasonable compute
budget), and so on. The execution layer then fails halfway through with a
worse error than catching this at plan time.

### Reproduction

1. Without installing bindcraft: ask the agent to plan a binder design with
   method=BindCraft. Plan is created and approved. `design.bindcraft` then
   fails at exec time with "tool not installed".
2. Plan an enzyme design with method=BindCraft (the agent will sometimes do
   this when it confuses the binder-side and enzyme-side stacks). Plan
   approves; downstream fails.

### Root cause

`internal/tools/plan/plan.go::PlanCreate` validates the schema shape but not
its semantic feasibility: no installed-tool check, no method↔application
compatibility table, no filter-range bounds.

### Proposed fix

1. **Tool-availability cross-check.** Plan reads the local backend's
   installed-tools list (post-Bug 13: `Installer.Status(name).Image != ""`)
   and rejects plans whose method tool isn't installed. Error:
   `method 'BindCraft' requires tool 'bindcraft' which is not installed —
   run /install bindcraft or pick a different method`.
2. **Filter-range bounds.** Schema-level: ipSAE ∈ [0, 1], pLDDT ∈ [0, 100],
   Kd > 0 (with unit normalisation), shortlist ∈ [1, 500] (soft cap; >500
   reports "exceeds compute budget — confirm or trim").
3. **Application↔method compatibility table.** Hard-coded `internal/tools/
   plan/compat.go` table:
   ```go
   var compat = map[Application][]Method{
       "binder":           {BindCraft, RFdiffusion},
       "enzyme_design":    {RFdiffusion, LigandMPNN},
       "antibody":         {RFantibody, RFdiffusion},
       // …
   }
   ```
   `plan.create` rejects incompatible combos with a message naming the
   compatible methods for the requested application.
4. **Test matrix.** Table-driven test covering each rejection path; a
   golden plan that passes all checks; a plan that initially fails the
   tool-availability check, then passes after a mocked installer marks
   bindcraft installed.

### Acceptance criteria

1. `plan.create` with method=BindCraft and no bindcraft image installed
   returns the named-tool error.
2. `plan.create` with filters out of range returns a per-field bound error
   listing the valid range.
3. `plan.create` with an incompatible application↔method pair returns the
   list of compatible methods for the requested application.
4. The compatibility table lives in `compat.go` (one source of truth) and
   is exercised by tests.

### Dependency note

Best landed after Bug 5 (citation validation) and Bug 13 (container backend
exposes installed-tool status). Pulled into Phase 2 of the agent allocation.

### Out of scope

- **Cost / time estimation** at plan time beyond what `plan.create` already
  computes. The cost model is unchanged.
- **Plan repair suggestions.** When `plan.create` rejects, it tells the user
  *what* is wrong but doesn't auto-rewrite the plan. v0.8+.

### Effort estimate

~half a day.

## Bug 12 — No shared retry/timeout/telemetry infrastructure for external calls

### Summary

Bugs 9 and 10 both need: retry-with-backoff, per-call timeouts, and
structured telemetry so failures are debuggable. Inlining that in two places
is wasteful; a third caller (UniProt in Bug 10, more in v0.8) makes it
copy-paste rot. v0.7 introduces a single `internal/transport/http.go`
wrapper so every external client (PMC, PDB, UniProt, …) shares the same
behaviour.

### Reproduction

N/A — this is infrastructure, not a user-visible bug. It exists as its own
bug entry because (a) it's a non-trivial code addition, (b) Bugs 9-10
depend on it, (c) tracking it as a separate workstream lets the foundation
land before the dependent bugs branch off master.

### Proposed fix

```go
// internal/transport/http.go
type Client struct {
    base       *http.Client
    retries    int
    backoff    func(attempt int) time.Duration  // default: 1s, 3s, 9s
    perCallTO  time.Duration                    // default: 30s
    onEvent    func(Event)                      // telemetry hook
}

type Event struct {
    Tool       string
    URL        string
    DurationMS int64
    Status     int
    Attempt    int
    Err        error
}

func New(opts ...Option) *Client { … }
func (c *Client) Do(ctx context.Context, req *http.Request, tool string) (*http.Response, error) { … }
```

The `onEvent` hook by default writes to `~/.proteus/logs/telemetry.jsonl`
(one event per line, JSON). Tests stub `onEvent` to capture calls. Each
external client (corpus.search PMC client, lab.targets PDB client, new
UniProt client) constructs one `transport.Client` with its tool name and
uses it for every external call.

### Acceptance criteria

1. `internal/transport/http.go` exists with the API above.
2. Unit tests cover: retry on 503-then-200, no-retry on 400, timeout
   abort, telemetry event emission.
3. `corpus.search` PMC client uses it (Bug 9 wiring).
4. `lab.targets_search` PDB and UniProt clients use it (Bug 10 wiring).
5. Telemetry file is written and has one valid JSON object per line; a
   subsequent run appends, doesn't truncate.

### Out of scope

- **A telemetry viewer / dashboard.** The JSONL is enough for grep / jq;
  v0.8+ can build a TUI surface.
- **Circuit breaker** for repeatedly-failing endpoints. Worth it later;
  not needed at this scale.

### Effort estimate

~3 hours. Mostly tests.

# Workstream C — GB10 GPU via containers

## Bug 13 — Local backend can't run sm_121-capable PyTorch on the GB10; per-tool venvs are the wrong shape

### Summary

The GB10 development box is sm_121. Stock PyPI PyTorch wheels don't ship
sm_121 kernels, so any tool that imports torch and calls into CUDA fails
on the GB10 even when the venv install succeeds. Building PyTorch from
source per tool venv is slow (~30 min each) and brittle. The right
architectural fix is to run every tool inside an NGC PyTorch container,
which NVIDIA pre-builds with sm_121 support and ships at `nvcr.io/nvidia/
pytorch:<tag>`. This also resolves bindcraft's conda-only PyRosetta
problem (PyRosetta installs cleanly into the image at build time, cached
once) and unifies the GPU story across the registry.

### Reproduction

1. `/install bindcraft` (after Bug 8's `requirements.txt` fix) completes.
2. Run a tiny bindcraft job. Fails with `CUDA error: no kernel image is
   available for execution on the device` (sm_121).
3. Same failure shape for any other tool calling torch.cuda.

### Root cause

`internal/backends/local/installer.go::Install` runs a sequence of host-side
bash commands inside `~/proteus/tools/<name>/` and creates a uv venv. Torch
in that venv is whatever's on PyPI — no sm_121.

### Proposed fix (architectural)

**Runtime detection.** New `internal/backends/local/runtime.go` exposes:

```go
type RuntimeKind int
const (
    RuntimeNone RuntimeKind = iota
    RuntimePodman
    RuntimeDocker
)

type Runtime struct {
    Bin  string       // absolute path to podman or docker
    Kind RuntimeKind
    HasGPU bool       // probed via `<bin> info` looking for nvidia
}

func Detect() (Runtime, error)
```

At startup, probe Podman first then Docker. If neither, surface a clear
"no container runtime" message via `/doctor` and refuse to install tools
that require GPU. (CPU-only tools — `ipsae` — could still run with a
fallback exec path, but for v0.7 simplicity, everything goes through the
container path.)

**Base image.** One constant in `internal/backends/local/registry.go`:

```go
const BaseImage = "nvcr.io/nvidia/pytorch:25.04-py3"  // verify tag pre-merge
```

Every tool's Containerfile FROM-s this. PyTorch lives in the base; per-tool
Containerfiles install only tool-specific Python deps on top.

**Per-tool schema in `tools.toml`.** The existing `install_steps` list is
replaced by:

```toml
[tools.proteinmpnn]
image_tag      = "proteus/proteinmpnn:v1.0.1"
containerfile  = "containerfiles/proteinmpnn.Containerfile"
entrypoint     = "python /opt/proteinmpnn/protein_mpnn_run.py"
gpu            = true
weights_paths  = ["/models/proteinmpnn/"]     # mounted at run-time
timeout_seconds = 1800
smoke_test     = "python /opt/proteinmpnn/protein_mpnn_run.py --help"
```

Containerfiles live under `internal/backends/local/containerfiles/<tool>.
Containerfile` and are embedded via `//go:embed`.

**Installer rewrite.** `Installer.Install()`:
1. `Runtime.Pull(BaseImage)` — once, cached after first run.
2. Write the Containerfile to a temp build dir.
3. `<runtime> build -t <image_tag> -f <containerfile> .`.
4. `Installer.Status(name)` returns `{Image string, Built bool}`. Built ⇔
   image present in `<runtime> images`.

**Runner rewrite.** `Runner.Run()`:
1. Compose the args list from the tool's `entrypoint` + the templated
   command (existing templating logic stays).
2. `<runtime> run --rm --name <jobID> --gpus all -v <workspace>:/work -v
   <models_cache>:/models -w /work <image_tag> <args>`.
3. Stream stdout/stderr to the job log (v0.6 Bug 2's plumbing).
4. Cancellation: `<runtime> kill <jobID>` instead of process-group kill
   (v0.6 Bug 7's logic adapts — still kills the underlying process tree
   because Podman/Docker handle that).

**Mount conventions.**
- Workspace: `<host workspace> → /work` (rw).
- Models cache: `~/.proteus/models/<tool>/ → /models` (rw). Survives image
  rebuilds and `/uninstall`.

**/doctor probe.** New section above "Local protein tools":

```
Container runtime
  ok    podman 4.9.4 (rootless)
  ok    nvidia-container-toolkit (CDI)
  ok    base image nvcr.io/nvidia/pytorch:25.04-py3 cached
```

If runtime is missing: `── no container runtime — install podman or docker`.
If NVIDIA toolkit missing: `!! nvidia-container-toolkit not detected — GPU
tools will not work`.

**Migration story.** v0.6 users with existing `~/proteus/tools/<tool>` venv
dirs: on first v0.7 run, `/doctor` flags each as `── legacy venv install
detected — run /install <tool> to migrate`. Old dirs are NOT auto-deleted
(user can `rm -rf` after verification).

**Models cache pre-population.** A shared helper `internal/backends/local/
models_cache.go::EnsureWeights(toolName string, urls []string) error`
downloads model weights to `~/.proteus/models/<toolName>/` on first
install. Used by every tool that needs weights (rfdiffusion, ligandmpnn,
boltz2, chai1, rfantibody).

### Acceptance criteria

1. `Runtime.Detect()` returns Podman when present, Docker as fallback,
   `RuntimeNone` if neither — unit test mocks `exec.LookPath`.
2. `Installer.Install("proteinmpnn")` on a clean machine: pulls base image
   once, builds the tool image, and `Installer.Status` reports
   `{Image: "proteus/proteinmpnn:…", Built: true}`.
3. `Runner.Run` invokes `<runtime> run --rm --gpus all -v <ws>:/work …`
   under the hood (asserted via a test seam that captures the command
   instead of executing it).
4. `/doctor` shows the new "Container runtime" section.
5. `/uninstall <tool>` removes the image (`<runtime> rmi <image_tag>`) but
   leaves `~/.proteus/models/<tool>/` (weights are expensive to re-download).
6. Legacy venv dirs trigger the migration message but are not deleted.
7. Cancelling a running job kills the container within 5s and the job log
   shows the SIGKILL trace.
8. End-to-end manual check on the GB10 (release acceptance): one tool
   (proteinmpnn) installs, runs a smoke test, and confirms
   `torch.cuda.is_available()` inside the container.

### Out of scope

- **Publishing pre-built images to GHCR.** Each user builds locally for
  v0.7. v0.8 may add a Proteus GHCR registry.
- **Multi-arch images.** v0.7 targets aarch64 (GB10) and x86_64 (laptop)
  but each user builds for their host arch only.
- **Container-in-container** for tools that already ship Docker recipes
  (e.g. some Baker lab tools have Dockerfiles upstream). v0.7 uses NGC
  PyTorch as the universal base regardless.
- **Rootless Docker.** Not actively supported; Podman rootless is the
  documented path.

### Effort estimate

~2-3 days for the platform agent. Containerfiles + per-tool work is
sized in Bugs 14-22, not here.
